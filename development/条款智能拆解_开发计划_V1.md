# AIX保险条款智能拆解开发计划 V1

## 1. 项目目标

建设一条面向保险产品条款的智能拆解链路，使系统能够基于原始条款 PDF，自动或半自动输出可入库的“标准项 + 标准值”结构化结果，并支持人工复核后写入正式产品库。

一期目标不是盲目全自动，而是实现以下业务效果：

1. 机器能够对高频标准项完成稳定预填
2. 人工拆解从“从零录入”变成“审核候选结果”
3. 全流程可追溯，能够回看原文证据、页码、抽取方法与置信度
4. 建立评测体系，为后续扩大到全量产品提供依据

## 2. 已知项目基础

当前已具备以下核心基础：

1. 标准项库：`cmb_coverage`
2. 标准值落库口径：`cmb_product_coverage.standard_content`
3. 临时数据库中已有 10 款重疾产品的标准拆解结果
4. 桌面 `10款重疾` 文件夹中已有对应原始条款 PDF
5. 新库已拆解产品超过 1800 款，老库和表格中还有数千款，且拥有原始文件

这意味着本项目可以采用“监督式抽取 + 人机协同复核”的路线，而不是从零探索。

## 3. 一期范围

一期只聚焦重疾险条款智能拆解，不覆盖全部险种。

### 3.1 输入范围

主输入：

1. 条款 PDF
2. 标准项库
3. 已拆解样本

辅助输入：

1. 产品说明书
2. 费率表
3. 利益演示

一期主链路默认只依赖条款 PDF，辅助材料仅用于校验或补证，不纳入核心抽取路径。

### 3.2 输出范围

一期输出为“待复核候选结果”，不直接默认写正式表。

输出内容包括：

1. 标准项候选
2. 标准值候选
3. 原文证据
4. 页码与文本块定位
5. 置信度
6. 抽取方式
7. 审核状态

### 3.3 一期优先字段

建议先覆盖 20 到 30 个高频标准项，优先包括：

1. 投保年龄
2. 保险期间
3. 交费期间
4. 等待期
5. 重疾赔付次数
6. 中症赔付次数
7. 轻症赔付次数
8. 身故责任
9. 豁免责任
10. 少儿特疾责任
11. 成人特定疾病责任
12. 可选责任
13. 保额限制
14. 重疾分组情况
15. 首次赔付比例与后续赔付比例

## 4. 一期成功标准

一期验收不以“是否全自动”为标准，而以“是否具备可用生产能力”为标准。

建议验收指标：

1. 高优先级字段覆盖率达到 80% 以上
2. 高频结构字段抽取准确率达到 90% 以上
3. 候选结果都能回溯到原文证据
4. 人工拆解平均耗时下降 50% 以上
5. 能区分自动通过、人工复核、无法识别三类结果

## 5. 技术路线

采用“规则召回 + 大模型理解 + 规则规范化 + 人工复核”的组合路线。

### 5.0 业务治理硬约束

以下约束为本项目实施红线：

1. 产品规则、定价、病种规则、数据库字段映射，必须来自统一数据源和统一规则，不允许由模型自行生成正式口径。
2. LLM 在本项目中只能输出“抽取建议”或“候选结果”，不能直接写正式业务库。
3. 不允许多个入口产生不同的正式结论，所有正式结果必须汇总到同一条规则校验与入库主链路。
4. 条款智能拆解能力只能增强现有标准拆解流程，不能绕过既有产品库口径。

### 5.1 规则优先处理

适合规则化的步骤：

1. PDF 解析
2. 页码、段落、表格切分
3. 标题识别与章节定位
4. 高结构字段抽取
5. 数值、区间、枚举值规范化
6. 冲突校验与格式校验
7. 过滤无效标准项，如“招行”前缀字段

### 5.2 大模型优先处理

适合大模型的步骤：

1. 将原文语义映射到标准项
2. 提取复杂责任描述
3. 跨段落合并理解
4. 输出证据归因
5. 识别未覆盖标准项或标准项定义不充分的问题

## 6. 分阶段计划

### 阶段 A：PoC 数据集与评测基线

目标：把 10 款重疾做成金标准验证集。

工作项：

1. 导出临时数据库中 10 款产品的标准拆解结果
2. 对齐原始 PDF 与产品 ID
3. 建立评测样本结构
4. 明确一期标准项白名单
5. 确定字段级评测口径

交付物：

1. 10 款金标准样本集
2. 字段白名单
3. 第一版评测报表模板

### 阶段 B：智能拆解引擎 MVP

目标：输出可复核的候选拆解结果。

工作项：

1. 文档解析模块
2. 文本块与表格块切分
3. 规则抽取器
4. 标准项映射器
5. 标准值规范化器
6. 质量校验器

交付物：

1. 单产品拆解接口
2. 批量拆解任务接口
3. 候选结果 JSON
4. 字段级准确率报表

### 阶段 C：人机协同复核

目标：将机器结果纳入实际业务流程。

工作项：

1. 候选结果入待审核表
2. 复核界面
3. 接受、修改、驳回操作
4. 审核结果回写
5. 错因标记与反馈闭环

交付物：

1. 复核工作台
2. 复核状态流转
3. 审核日志

### 阶段 D：高置信自动化扩展

目标：对稳定字段开放自动通过。

工作项：

1. 建立字段级阈值
2. 自动通过策略
3. 失败回退策略
4. 按险种扩展模板

交付物：

1. 自动通过规则集
2. 风险控制策略
3. 险种扩展计划

## 7. 角色分工建议

你作为项目负责人兼开发负责人，建议按以下方式组织：

1. 你：项目节奏、标准项治理、验收口径、关键设计决策
2. Claude code：方案监督、文档审查、实现质量校验、测试清单把关
3. 开发同事 A：文档解析、任务调度、接口与数据流
4. 开发同事 B：复核工作台、状态流转、入库链路
5. 精算/产品拆解同事 A：标准项白名单、字段定义、值规范化口径
6. 精算/产品拆解同事 B：10 款样本复核、错因分类、评测校对

如果后续增加人手，优先增加两类角色：

1. 负责历史样本整理与对齐的数据工程支持
2. 负责复核流程落地的业务运营支持

## 8. 当前最优先任务

按优先级建议立刻启动以下事项：

1. 明确一期标准项白名单
2. 导出 10 款产品临时库金标准结果
3. 明确产品 ID 与 PDF 文件映射关系
4. 定义候选结果中间表结构
5. 定义字段级评测方法
6. 完成文档解析能力选型

## 9. 风险与控制

主要风险：

1. PDF 解析质量不稳定
2. 标准项定义过粗或过细，导致映射不稳
3. 不同公司条款表达差异过大
4. 样本多但标签质量不完全一致
5. 业务方过早追求全自动入库

控制策略：

1. 一期先做重疾险，不全险种铺开
2. 一期先做高频字段，不追求全字段
3. 先候选后复核，不直接写正式表
4. 所有结果保留证据与审计日志
5. 按字段类型分别评估，不用单一总分掩盖问题
6. 所有正式结果统一从单一入库链路进入，避免双轨结论

## 10. 交付检查清单

每次开发、修复、提测、交付前，必须逐项确认以下问题都能回答“是”：

1. 同一业务规则是否只有一个实现，没有双轨逻辑。
2. 数据模型是否支持真实业务，而不只是演示场景。
3. 核心逻辑是否集中、可复用、可测试，没有散落在 UI 回调里。
4. 修一处时是否同步覆盖相关口径，没有遗漏另一处。
5. 修改后是否同步删除无用代码，没有叠加旧逻辑。
6. 文档、实现、导出、测试是否同步更新。
7. 主链路是否有真实测试覆盖，而不只是演示页通过。
8. 结果是否可解释、可复核、可追溯。
