# AIX保险条款智能拆解开发文档 V0.2

代码分支：`development/clause-parse`
文档解析工具：`doc_to_md`（[https://github.com/ferryhe/doc_to_md](https://github.com/ferryhe/doc_to_md)）

版本说明：

1. `V0.1`：完成条款主链路、盲测 PoC、费率处理后结果接入和基础治理设计
2. `V0.2`：补充多来源输入边界、共享契约、反向生成定位，以及人工复核页对“来源 + 判定链路 + 逐层诊断链”的强要求

## 1. 目标与定位

本能力用于将原始资料包中的条款、费率、产品说明书等重要信息自动或半自动拆解为可入库的标准化结果，输出口径与现有产品库一致，即：

1. 标准项来自 `cmb_coverage`
2. 条款类标准值最终落入 `cmb_product_coverage.standard_content`
3. 费率、病种等其他重要信息也需进入对应标准库表，不允许长期停留在临时文件或候选结果层

本能力的最终定位不是只做条款字段预填，而是将条款、费率、产品说明书等重要业务信息统一标准化并进入标准库；当前阶段先从条款层和高频结构字段起步，逐步扩展到责任类、费率类和其他资料来源。

从智能保险体整体能力看，本项目不仅服务“读懂既有产品”，也服务未来“反向生成产品文档”的能力建设。也就是说，当前沉淀的标准项、标准值、来源逻辑、公司习惯和规范化映射，未来应能支持保险公司按既定责任结构和标准值，反向生成符合其惯用表达的条款、说明书和相关材料初稿。

本能力的当前实现阶段边界为：

1. 当前代码主链路先完成条款拆解与核心结构字段抽取
2. 费率表、产品说明书、病种、现价等能力后续分阶段建设
3. 所有进入标准库的重要信息都必须有来源，不允许无来源入库
4. 当前未建成原始费率解析能力前，可在实验中调用处理后费率验证整体拆解能力，但不能将其等同于正式费率解析能力完成
5. 条款或其他资料信息不足时只能输出不可完整抽取或进入复核，不能猜测正式值
6. `处理后费率表 -> cmb_product_rate` 的标准化读取层已完成首个生产口径验证，可作为费率入库链路的已验证基础模块

## 2. 总体架构

整体流程如下：

1. 原始资料包接入
2. 文档解析
3. 文本块与表格块标准化
4. 多来源字段候选召回
5. 大模型标准项映射与复杂值抽取
6. 标准值规范化
7. 质量校验
8. 候选结果落中间表
9. 人工复核
10. 正式入库到标准库

## 3. 核心设计原则

1. 不直接把模型输出写入正式表
2. 所有候选结果必须带证据
3. 由易到难是开发节奏，不是目标边界；最终要覆盖全量重要标准项
4. 规则与模型分层，不混写
5. 每次抽取都要保留模型版本、规则版本、任务版本
6. 排除当前无效使用范围的“招行”前缀字段
7. 条款、费率、产品说明书等重要信息最终都要进入标准库

### 3.1 业务治理硬约束

1. 产品规则、定价、病种规则、数据库字段映射，必须来自统一数据和统一规则，不能由模型自行生成正式口径。
2. LLM 只能给出抽取建议、候选标准项、候选标准值和证据归因，不能直接写正式库。
3. 不允许批量任务入口、人工复核入口、后续 API 入口各自产生不同正式结论，必须统一汇总到同一套校验和入库主链路。
4. 正式业务结果只能以统一规则校验后的入库结果为准，模型结果不构成业务真相。

### 3.2 交付验收检查

每次开发或修复交付前，必须确保以下问题全部可回答“是”：

1. 同一业务规则只有一个实现，无双轨逻辑。
2. 数据模型支持真实业务，不只是演示场景。
3. 核心逻辑集中、可复用、可测试，不散落在 UI 回调里。
4. 修一处不漏另一处，相关口径已经同步。
5. 修改后已同步删除无用代码，没有叠加旧逻辑。
6. 文档、实现、导出、测试四者同步。
7. 主链路有真实测试覆盖，不只是演示页。
8. 结果可解释、可复核、可追溯。

## 4. 输入设计

### 4.1 输入对象

主输入：

1. 产品条款 PDF
2. 原始费率表
3. 产品说明书
4. 标准项库
5. 已拆解样本
6. 已处理标准费率、病种等数据库结果

说明：

1. 产品真实拆解应以原始资料包为输入，而不是单一条款文件
2. 字段可按来源矩阵分别依赖条款、费率表、产品说明书或其他已标准化结果
3. 当前代码实现先以条款主链路为基础，逐步接入费率和说明书能力
4. 在盲测实验中，允许调用处理后费率验证整体拆解能力，但输出必须记录来源类型和来源文件

### 4.2 多来源输入结构

```json

    ],
    "raw_rate": [
      {
        "document_id": "doc_rate_001",
        "document_type": "rate_pdf",
        "file_name": "700-招商信诺真惠保重大疾病保险（互联网专属）-费率表.pdf"
      }
    ],
    "processed_rate": [
      {
        "document_id": "doc_rate_processed_001",
```

说明：

1. 正式目标是支持“原始资料包 + 已处理标准化结果”的多来源输入。
2. 当前盲测阶段允许调用处理后费率，但这只用于验证整体拆解能力，不代表原始费率解析能力已经完成。
3. 任一字段若使用了处理后费率或其他已标准化结果，输出中必须记录 `source_type`、`source_file`、`evidence`。

### 4.3 文档输入结构

```json
{
  "product_id": "1134",
  "product_name": "国华康佑保B终身重疾计划",
  "document_type": "clause_pdf",
  "document_id": "doc_1134_clause_pdf_001",
  "source": "desktop_sample_set",
  "batch_id": "batch_20260324_01"
}
```

说明：

1. 文件上传通过独立接口 `POST /api/v1/clause-parse/upload` 完成。
2. 上传接口返回 `document_id`。
3. 在单文档模式下，后续拆解任务只传 `product_id + document_id`。
4. 在资料包模式下，任务应支持传入 `document_package_id` 或多来源 `document_id` 列表。

## 6. 模块设计

### 6.1 文档解析模块

职责：

1. 读取 PDF
2. 识别页码
3. 切分段落、标题、表格
4. 输出标准文本块

输出对象建议：

```json
{
  "page": 4,
  "block_id": "p4_b12",
  "block_type": "paragraph",
  "title_path": ["第一部分", "投保规则"],
  "text": "本合同接受的投保年龄为出生满28天至50周岁。"
}
```

说明：

1. 文档解析调用 `doc_to_md` API：`POST /apps/conversion/convert`，`engine=markitdown`
2. 输出 Markdown 中间文件，命名规则：`{product_id}_{document_type}.md`
3. 中间文件存放目录：`~/code/development/data/md_cache/`
4. 扫描型 PDF 一期不支持，直接标记为 `cannot_extract` 并提示人工处理

### 6.2 规则候选召回模块

职责：

1. 对结构化字段做高召回初筛
2. 给大模型缩小判断范围

适合规则处理的字段：

1. 投保年龄 `[rule]`
2. 保险期间 `[rule]`
3. 交费期间 `[rule]`
4. 等待期 `[rule]`
5. 赔付次数（重/中/轻） `[rule]`
6. 金额、比例、区间表达 `[rule]`
7. 身故/豁免/特定疾病责任 `[rule+llm]`
8. 可选责任 `[llm]`
9. 重疾分组情况 `[llm]`
10. 首次/后续赔付比例 `[rule+llm]`

规则输出：

1. 候选文本块
2. 命中规则
3. 初步值
4. 初步置信度

补充说明：

1. 部分基础字段虽然属于高频字段，但在部分条款中仅写“以保险单载明为准”或“由投保时约定并在保险单上载明”
2. 对这类字段，本模块允许输出候选证据，但不允许在条款信息不足时补造正式值
3. 若条款中无法形成完整标准值，应输出 `cannot_extract_from_clause` 或进入 `review_required`

### 6.3 标准项映射模块

职责：

1. 将文本块映射到标准项
2. 解决同义表达与长句条件
3. 判断当前表达是否能落入现有标准项

输入：

1. 文本块
2. 标准项白名单
3. 历史样本参考

输出：

1. `coverage_id`
2. `coverage_name`
3. 映射理由
4. 置信度
5. 是否存在候选冲突

### 6.4 标准值规范化模块

职责：

1. 将原文值转成入库口径
2. 统一数值、区间、次数、枚举表达

示例：

1. `出生满28天至50周岁` -> `0(28天)-50周岁`
2. `等待期为180日` -> `180天`
3. `给付3次，每次100%基本保险金额` -> 保留标准化主值，并记录备注文本

说明：

复杂责任不建议在一期过度压缩为单值，必要时保留：

1. 主值
2. 补充描述
3. 原文证据

补充边界：

1. 规范化仅针对条款内已明确给出的值
2. 对于费率、产品说明书等其他来源字段，应在对应来源能力建成后按统一规则规范化入库
3. 不允许跨来源猜测值；只能基于已确认的来源证据形成标准值
4. 对于当前实现阶段尚未接入的来源字段，只能保留证据并进入复核

### 6.5 质量校验模块

职责：

1. 校验字段格式
2. 校验同字段多值冲突
3. 校验字段之间的一致性
4. 校验是否命中无效标准项

重点校验项：

1. 招行前缀字段不输出到一期有效结果
2. 同一标准项若多处命中，需标记冲突
3. 数值与区间必须通过格式规范化
4. 复杂责任若无法唯一表达，强制进入人工复核
5. 条款中若仅存在原则性描述、未给出完整标准值，必须标记为 `cannot_extract_from_clause`

### 6.6 复核入库模块

职责：

1. 展示候选结果
2. 支持接受、修改、驳回
3. 保留审核日志
4. 审核通过后写入正式表

复核页面最低要求：

1. 可看原文证据
2. 可看页码
3. 可看模型/规则结果
4. 可修改规范化值
5. 可记录错因
6. 可看字段来源类型与来源文件
7. 可看“原始文件值 -> md值 -> block值 -> 标准值”的逐层链路
8. 可看判定链路，包括来源优先级、命中规则、规范化步骤和最终入库值

补充说明：

1. 复核页面不是只看最终结果，还要能诊断错误发生在“原始文件 -> md -> block -> 抽取 -> 规范化”哪一层。
2. 原文来源值既服务业务复核，也服务技术排障，用于验证原始文件、Markdown 转换、切块和抽取链路是否一致。
3. 页面应同时支持业务审单和技术诊断两种视角，默认可简化展示，但字段展开后必须能看到完整判定链路。

## 9. 规则与大模型分工

建议按以下原则执行：

规则负责：

1. 文档切分
2. 强结构字段召回
3. 值规范化
4. 一致性校验
5. 白名单与黑名单控制

大模型负责：

1. 标准项语义映射
2. 复杂责任理解
3. 多段合并
4. 证据归因
5. 未覆盖项识别

## 10. 人机协同策略

禁止一期直接追求全自动。

建议按置信度与字段类型分流：

1. 高频结构字段 + 高置信 + 校验通过：可进入 `auto_pass`
2. `auto_pass` 一期阈值：`confidence >= 0.92`，且字段在规则白名单内
3. 高频结构字段但存在冲突：进入 `review_required`
4. 复杂责任字段：默认 `review_required`
5. 无法映射或标准项不充分：进入 `cannot_extract`

补充约束：

1. `auto_pass` 只表示自动通过到候选确认链路，不代表模型可绕过统一入库规则。
2. 正式库写入必须经过统一校验服务，不允许由模型调用链直接提交。
3. 人工复核时必须能同时查看“结果、来源、判定链路”，不允许只展示最终标准值。
4. 对多来源字段，复核界面必须明确展示来源优先级和最终采用哪一份材料，避免人工误判。

## 11. 评测设计

一期建议先以 10 款重疾样本为金标准验证集。

评测指标：

1. 标准项命中率
2. 标准值准确率
3. 标准值规范化一致率
4. 证据可追溯率
5. 冲突识别率
6. 人工复核耗时下降比例
7. 重要信息入库完整率

建议按字段分层统计：

1. 基础结构字段
2. 半结构字段
3. 复杂责任字段

## 12. 一期实施顺序

### 第一阶段：条款层打底

1. 明确标准项白名单与字段来源矩阵
2. 对齐 10 款样本的金标准结果
3. 打通条款 PDF 解析与文本块输出
4. 完成 Tier A 高结构字段规则抽取
5. 基于盲测结果修复条款层规则缺口

### 第二阶段：费率与多来源补足

1. 接入费率表抽取层或处理后费率读取层，补足基础字段
2. 明确条款、费率、产品说明书之间的字段优先级和冲突规则
3. 完成中间表与多来源结果存储
4. 完成复核工作台与统一入库链路
5. 其中“处理后费率读取层”已在 `1010003676` 上完成生产库零差异验证，应优先作为费率标准化入库的第一落地点

### 第三阶段：责任类扩展

1. 接入大模型完成复杂责任项映射
2. 扩展到重疾责任类、轻症/中症类、特定责任类字段
3. 做 10 款样本验收后扩展到 100 款样本
4. 责任类字段属于必须覆盖范围，不可长期搁置在“一期之后”

## 13. 已知风险

1. PDF 文本质量参差不齐
2. 历史拆解数据口径可能不完全一致
3. 部分标准项定义可能无法承载真实复杂条款
4. 复杂责任难以压缩为单一标准值
5. 多资料之间可能出现来源冲突，需统一优先级和复核规则
6. 当前条款层、费率层、说明书层建设进度不一致，短期内会出现能力不均衡
