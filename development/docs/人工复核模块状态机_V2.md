# 人工复核模块状态机 V2

**版本：** V2.0
**关联：** 人工复核工作台_V2_产品级设计.md，人工复核信息架构_V2.md

---

## 一、状态机概览

```
任务级状态机（TaskStatus）
    ↕
字段级状态机（CoverageItemStatus）
    ↕
入库状态层（ImportStatus）
```

三层状态机相互联动：字段级变更影响任务级；任务级决定是否允许入库；入库状态层负责提交、成功、失败回流。

---

## 二、任务级状态机

### 2.1 状态定义

| 状态 | 含义 |
|---|---|
| `pending_review` | 已创建，等待复核员认领 |
| `in_review` | 复核员已打开，正在进行中 |
| `review_completed` | 所有字段已裁决，任务完成 |
| `returned_for_materials` | 打回，需补充资料 |
| `import_submitted` | 复核完成后已提交入库请求 |
| `import_failed` | 入库失败（后端校验未通过） |
| `archived` | 已归档（入库成功或手动归档） |

### 2.2 状态转移规则

```
              [认领/打开任务]
pending_review ─────────────────→ in_review
                                       │
                     ┌─────────────────┤
                     ↓                 ↓
           [打回补资料]         [所有字段裁决完成]
    returned_for_materials         review_completed
                     │                 │
         [补完重新提交]          [提交入库]
                     ↓                 ↓
                 in_review      import_submitted
                                       │
                            ┌──────────┤
                            ↓          ↓
                     import_failed   archived
                            │
                    [修正后重提交]
                            ↓
                     import_submitted
```

### 2.3 任务级转移条件

| 从 | 到 | 条件 |
|---|---|---|
| `pending_review` | `in_review` | 复核员打开任务 |
| `in_review` | `review_completed` | 所有 CoverageItem 的 status ∈ {accepted, modified, rejected, cannot_extract, cannot_extract_from_clause} |
| `in_review` | `returned_for_materials` | 复核员主动操作，须填写打回原因 |
| `returned_for_materials` | `in_review` | 资料补充完毕，重新分配 |
| `review_completed` | `import_submitted` | 复核员点击"提交入库" |
| `import_submitted` | `archived` | 入库服务确认成功 |
| `import_submitted` | `import_failed` | 入库服务校验未通过，返回错误 |
| `import_failed` | `import_submitted` | 复核员修正后重提交 |

## 三、字段级状态机

### 3.1 状态定义

| 状态 | 含义 | 是否阻断入库 |
|---|---|---|
| `review_required` | 系统识别后置为待复核（候选值存在，需人工确认） | 是（未裁决不允许任务完成）|
| `candidate_ready` | 系统自动判定置信度高，可人工快速接受 | 是（仍需人工确认）|
| `accepted` | 人工接受系统候选值 | 否 |
| `modified` | 人工修改后接受 | 否 |
| `rejected` | 人工驳回，标注错因，阻断入库 | 是 |
| `cannot_extract` | 无候选值，资料中不存在该字段 | 视情况（需标注是否允许入库）|
| `cannot_extract_from_clause` | 字段存在但条款无此项（产品特性决定，如少儿产品无交费频率月交）| 否（属正常业务逻辑）|
| `pending_materials` | 需补充资料才能裁决，当前暂挂 | 是 |

### 3.2 状态转移规则

```
                    ┌──── 系统提取后 ────┐
                    ↓                   ↓
             review_required      candidate_ready
                    │                   │
        ┌───────────┼───────────────────┤
        ↓           ↓                   ↓
    [接受]       [修改后接受]         [驳回]
    accepted       modified           rejected
                                        │
                                [重新复核]
                                        ↓
                                  review_required
        ↓
    [标记无法提取]
    cannot_extract
    cannot_extract_from_clause

        ↓
    [标记待补资料]
    pending_materials
        │
    [补完后复核]
        ↓
    review_required
```

### 3.3 字段级转移条件

| 从 | 到 | 条件 |
|---|---|---|
| `review_required` | `accepted` | 人工点击"接受"，final_value = normalized_value |
| `review_required` | `modified` | 人工修改 final_value 后点击"接受" |
| `review_required` | `rejected` | 人工点击"驳回"，reason_code 必填 |
| `review_required` | `cannot_extract` | 人工标注无法提取 |
| `review_required` | `cannot_extract_from_clause` | 人工标注条款无此项 |
| `review_required` | `pending_materials` | 人工标注需补资料 |
| `candidate_ready` | 同上各项 | 同上 |
| `rejected` | `review_required` | 人工选择重新复核 |
| `accepted` | `review_required` | 人工选择撤销接受 |
| `modified` | `review_required` | 人工选择重新修改 |
| `pending_materials` | `review_required` | 资料补充完毕，重置 |

## 四、入库决策层

### 4.1 入库可行性规则

系统在"提交入库"时校验：

```
任务可提交入库 当且仅当：
  task_status == "review_completed"
  AND
  所有 items 的 status ∈ {accepted, modified, cannot_extract_from_clause}
  AND
  NOT 存在 item.block_import == true AND item.status == rejected
```

**注：** `cannot_extract` 字段需人工在裁决时标注 `block_import`:
- `false`：允许该字段缺失进入入库（将入库为 null）
- `true`：该字段缺失阻断入库（需补充资料或人工填写）

### 4.2 入库字段映射

| 字段状态 | 入库内容 |
|---|---|
| `accepted` | `final_value` = `normalized_value`（系统值）|
| `modified` | `final_value` = 人工修改值 |
| `cannot_extract` + block_import=false | `standard_content` = null，`source_type` = "not_found" |
| `cannot_extract_from_clause` | `standard_content` = null，`source_type` = "not_applicable" |
| `rejected` | 不入库，记入审核日志 |

## 五、第三层：ImportStatus

这一层用于明确区分“已完成复核”和“已成功入库”。

### 5.1 状态定义

| 状态 | 含义 |
|---|---|
| `not_submitted` | 尚未提交入库 |
| `submitted` | 已发起入库请求 |
| `succeeded` | 入库成功 |
| `failed` | 入库失败，需回流复核 |

### 5.2 转移规则

```
not_submitted -> submitted -> succeeded
                        └-> failed -> not_submitted
```

### 5.3 约束

1. `TaskStatus=review_completed` 不等于已入库
2. `ImportStatus=failed` 时，任务应回到可修正状态
3. 前端提交按钮必须读后端 `ImportStatus`，不能只靠本地状态判断

## 六、任务级与字段级联动规则

| 条件 | 任务级变化 |
|---|---|
| 存在 status=`review_required` 的字段 | 任务进度显示"未完成" |
| 存在 status=`pending_materials` 的字段 | 任务级"完成"按钮置灰 |
| 存在 status=`rejected` + block_import=true 的字段 | 提交入库按钮置灰 |
| 所有字段都是终态 | 任务级可转为 `review_completed` |
| 依赖字段组内存在冲突 | 系统弹出依赖警告，不阻断但提醒 |

## 七、依赖关系触发规则

### 6.1 重疾责任链

```
触发条件：
  重疾赔付次数.final_value 变更时

规则：
  IF 次数 == "1次"
    AND 重疾分组.final_value NOT IN {"不涉及", "不分组", null}
    THEN 警告："次数=1时，分组应为'不涉及'"

  IF 次数 IN {"2次", "3次", ...}
    AND 重疾分组.status == "cannot_extract"
    THEN 提示："多次赔付产品，请确认分组设置"

  IF 重疾数量 < 100
    AND 次数 > "1次"
    THEN 弱警告："多次赔付产品重疾数量通常≥100，请确认"
```

### 6.2 投保年龄-天数链

```
触发条件：
  投保年龄 winning source == "processed_rate"
  AND final_value 匹配 ^0-\d+周岁$

规则：
  THEN 系统提示："当前年龄来自费率表，无天数说明，请从条款或说明书确认"
  行为：不阻断，但标记 requires_manual_days_confirmation = true
```

### 6.3 交费三联

```
触发条件：
  交费期间.final_value 变更时

规则：
  IF 交费期间仅包含"趸交"
    AND 交费频率 NOT IN {"趸交", null}
    THEN 警告："趸交产品，交费频率应为趸交或空"
```

## 八、操作权限与审计

### 7.1 操作记录（所有状态变更均记录）

```json
{
  "log_id": "...",
  "task_id": "...",
  "candidate_id": "...",
  "action": "accept | modify | reject | mark_missing | revert",
  "from_status": "review_required",
  "to_status": "accepted",
  "operator": "ops_user_01",
  "operated_at": "2026-03-26T10:30:00",
  "final_value": "2次",
  "reason_code": null,
  "review_comment": null
}
```

除状态变更日志外，V2 还应统一沉淀事件日志，用于串联“裁决 -> 任务完成 -> 提交入库 -> 入库结果”全过程：

```json
{
  "event_id": "evt_001",
  "task_id": "task_20260326_001",
  "candidate_id": "cov_age",
  "event_type": "decision_submitted",
  "from_status": "review_required",
  "to_status": "modified",
  "operator": "ops_user_01",
  "operated_at": "2026-03-26T10:30:00",
  "payload": {
    "final_value": "0（28天）-65周岁",
    "reason_code": "source_missing_need_manual_fix"
  }
}
```

### 8.2 撤销规则

- 字段级：任何终态（accepted/modified/rejected/cannot_extract）均可撤销回 `review_required`
- 任务级：`review_completed` 可回退到 `in_review`（在入库提交前）
- `import_submitted` 之后不允许撤销（需走退回流程）

## 九、前端状态管理要求

V2 前端应使用**明确的状态机管理**（非 V1 的 localStorage 临时存储）：

| 要求 | 说明 |
|---|---|
| 状态持久化 | 裁决动作实时 POST 到后端（或每次"保存裁决"时），不依赖浏览器缓存 |
| 乐观更新 | UI 立即响应裁决操作，后端失败时回滚 |
| 冲突检测 | 依赖规则在前端即时触发，不等待后端 |
| 进度计算 | 从 items[].status 实时计算，不存本地 |
| 测试友好 | 所有状态变更通过 action dispatch，可单测不依赖 DOM |
