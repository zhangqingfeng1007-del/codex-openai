## 二、已掌握的关键业务知识

### 2.1 拆解是多来源协同，不是单文档行为

每个字段的来源有优先级，不同字段、不同公司惯性不同：

```
条款 > 产品说明书 > 费率表 > 投核保规则 > 销售材料
```

- 投保年龄的"出生满XX天"：费率表永远不写（新华例外）
- 交费频率：条款会写，费率表尾部附注也会写，若都没有再从交费期间推断
- 公司材料有惯性：太保的等待期常隐藏在责任段落里，平安犹豫期习惯写20天

### 2.2 "不支持"不是无来源，而是负向确认

条款中未出现某权益约定 ≠ 无法判断。

- 来源 = "条款保单权益章节，未见相关约定"
- 结论 = "不支持"（负向来源确认）
- 这是保险拆解里的标准人工逻辑，需要专门建模

**负向确认规则（已形成）：**
> 对按业务惯例应核查的权益字段（保单贷款、减额交清、转换权等），若复核员通读主来源文件后未见相关约定，可将 final_value 填"不支持"，状态置 accepted，来源记录"经人工通读条款，未见相关权益约定"。

### 2.3 标准值不是字面抽取，是业务判定

- 候选值"10年，20年，30年"和标准值"10/20/30年交"语义等价，但格式不同
- 需要**规则判定层**做格式归一，而不是直接用字符串比较
- 这一层当前缺失，是最大技术缺口

### 2.4 覆盖项治理需要扩展机制

- 保险行业持续发展，新产品会出现已知标准值列表外的值
- 处理原则：不阻断，标记 new_value，人工确认后纳入 catalog
- 新值治理：≥3款产品出现 → 管理员审批 → 写入 JSON → PATCH 版本递增

### 2.5 拆解能力的终局是双向的

- 正向：原始文件 → 结构化字段 → 入库
- 反向：标准值 + 公司风格 → 生成条款 / 说明书 / 销售材料
- 当前结构化规则文档已是反向生成的基础素材

## 三、当前存在的问题

### P0（最大缺口）：规则判定层缺失

候选值与标准值之间存在格式差异，当前直接字符串比较，导致：
- 交费期间命中率 0%（格式 mismatch，非语义错误）
- 保险期间被误判为 new_value（"10年，20年，30年" vs "10/20/30年"）

**需要建立：** `normalize_value(coverage_name, raw_value) → standard_value`
**参考：** `docs/核心系统拆解规则_结构化拆分方案_V1.md`（Tier A 10字段已结构化）

---

### P1：投保年龄来源缺口

- 命中率 29%，主因是条款缺"出生满XX天"说明
- 多来源优先级逻辑（条款→说明书→费率→核保规则）未实现
- 当前只用条款单一来源

---

### P2：review task builder 仍是 Step 1

- 候选来源：`blind_test_results_v1.json`（LLM盲测），而非正式规则抽取输出
- 两个来源字段结构有差异，合并逻辑未建立
- 仅有 1010003851 一款产品的任务 JSON

---

### P3：前端标准值参考未显示

- `coverage_standard_values_v1.json` 已就绪
- 但前端复核页字段详情区还没展示"已知标准值参考列表"
- 复核员目前只能靠经验判断候选值是否合理

---

### P4：前端仍是原型，缺正式后端链路

- 提交复核结果目前是 mock（console.log）
- 缺：真实后端提交 API / 补录 catalog 搜索 / coverage 新增申请流程
- 缺：多人复核、任务分配、复核历史

---

### P5：coverage template 还是最小模型

- 当前覆盖 28 项（whitelist），全量 96 项治理体系未建立
- 计划/子计划、附加险继承关系、多条款合并产品未支持

## 四、风险点

| 风险 | 说明 | 缓解措施 |
|---|---|---|
| DB 不可直连 | 阿里云 IP 白名单限制，full_coverage_dump 是快照 | 需定期更新 dump，或申请白名单 |
| 候选来源混用 | blind_test 和 tier_a_rule_candidates 字段结构不统一 | 统一抽取层输出 schema，再接入 builder |
| 格式规范化错误引入 mismatch | normalize_value 错误会导致正确候选被误判 | 先建规则，人工 case-by-case 验证，再放量 |
| 新值被误判为标准值 | 业务快速发展，标准值库有滞后 | 定期 review new_value 记录，执行治理流程 |
| 前端与后端契约漂移 | 原型快速迭代，schema 可能与 V2 定义偏移 | 以 review_task_json_schema_v2.md 为唯一标准 |
