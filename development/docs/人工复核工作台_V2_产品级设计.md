## 一、核心问题重述

V1 的错误在于以"字段候选条目"为复核对象。
这让复核员看不到：

- 整产品资料包完整性
- 同一字段在多处来源的一致性
- 字段间的依赖关系（重疾次数↔分组等）
- 真正的入库风险项

V2 将复核对象升级为 **产品**，字段复核是产品复核的下一层。

---

## 二、两层复核结构

```
产品级复核（Task 粒度）
  └── 字段级复核（Candidate 粒度）
```

V2 页面主对象应明确为 `ReviewTask(Product)`，而不是松散的“产品元数据 + 字段列表”：

```json
{
  "task": {
    "task_id": "task_20260326_001",
    "task_status": "in_review",
    "rule_version": "rule_v2"
  },
  "product": {
    "product_id": "1010003851",
    "product_name": "招商信诺真惠保重大疾病保险（互联网专属）",
    "company_name": "招商信诺",
    "aix_category_id": 6001
  },
  "document_package": {
    "document_package_id": "pkg_1010003851_001",
    "files": []
  }
}
```

### 2.1 产品级复核

复核员先看整产品，决策：
- 资料包是否完整
- 是否存在高风险字段（冲突 / 来源缺失）
- 是否允许进入字段级复核
- 是否需要打回索取补充资料

**产品级状态机：** `pending_review` → `in_review` → `review_completed` | `returned_for_materials`

### 2.2 字段级复核

在产品级通过后，对每个标准项逐一/批量裁决：
- 接受 / 修改 / 驳回
- 标注错因 / 来源缺失
- 标注是否允许入库

**字段级状态机：** `review_required` → `accepted` | `modified` | `rejected` | `cannot_extract` | `cannot_extract_from_clause`

## 四、各区详细设计

### 区域 1：顶部导航栏

| 元素 | 内容 |
|---|---|
| 产品名 | `task.product_name` |

---

### 区域 2：标准项清单区（左栏）

**不是单条展示，而是按分组批量显示所有标准项。**

#### 分组方式

```
基础规则字段（Tier A）
  ├── 投保年龄            [已接受]
  ├── 保险期间            [已接受]
  ├── 交费期间            [待复核] ⚠
  ├── 交费频率            [待复核]
  ├── 等待期              [已接受]
  ├── 宽限期              [已接受]
  └── 犹豫期              [已接受]

责任类字段（Tier A）
  ├── 重疾赔付次数         [待复核] ⚠
  ├── 重疾分组             [待复核] ← 依赖重疾赔付次数
  └── 重疾数量             [已接受]

费率类字段（Tier A，来源 rate_table）
  ├── 投保年龄（费率侧）    [conflict] 🔴 ← 与条款侧冲突
  ├── 交费期间（费率侧）    [已接受]
  └── 交费频率（费率侧）    [缺失]

关联组：重疾责任链
  ├── 重疾赔付次数
  ├── 重疾分组
  └── 重疾数量
  （三个字段须联动审核，系统提示依赖关系）

缺失字段
  └── 特定疾病数量          [cannot_extract] — 条款/说明书均未找到
```

除了普通分组，V2 还应把依赖关系单独建模，避免前端只靠提示文案做人脑联动：

```json
{
  "dependency_group_id": "ci_chain",
  "group_name": "重疾责任链",
  "members": [
    "重疾赔付次数",
    "重疾分组",
    "重疾数量"
  ],
  "rules": [
    "次数=1次 -> 分组通常为不涉及",
    "次数>1次 -> 分组应为不分组或N组"
  ]
}
```

前端选中某字段时，同一 `dependency_group` 成员应同步高亮。

#### 每行显示内容

```
[状态 badge]  字段名              候选值（截断）       来源数量
⚠ 待复核      重疾赔付次数         2次                  clause × 1
🔴 冲突        投保年龄            条款:0(28天)-59 | 费率:0-59   2
✅ 已接受      保险期间            终身                  clause × 1
⬜ 缺失        特定疾病数量        —                     0
```

#### 交互
- 点击任意行 → 中栏展示该字段的多来源证据
- 点击分组标题 → 折叠/展开分组
- 右键 / 快捷键批量动作：批量接受同分组已就绪字段


### 区域 4：判定链路区（中栏下半）

对当前字段展示"从原始 → 入库"的完整推导链路：

```
来源优先级评估
  条款（优先级最高）  → 命中  → 0（28天）-59周岁
  说明书              → 命中  → 0（28天）-59周岁
  费率表              → 命中  → 0-59周岁 （缺天数，正常）

规范化
  原始：0（28天）-59岁
  → 统一格式：0（28天）-59周岁

最终入库字段映射
  coverage_name：投保年龄
  standard_content：0（28天）-59周岁
  source_type：clause
```

## 五、关联字段组（依赖关系）

以下字段须在同一视图内联动显示，不能孤立审核：

| 关联组 | 字段 | 依赖关系 |
|---|---|---|
| 重疾责任链 | 重疾赔付次数、重疾分组、重疾数量 | 次数=1→分组=不涉及；次数>1→需确认是否分组 |
| 年龄-来源链 | 投保年龄、来源文件类型 | age_min=0时须有天数，来源须是条款或说明书 |
| 交费三联 | 交费期间、交费频率、费率来源 | 费率表中交费期列表须与条款一致 |
| 少儿责任链 | 特定疾病数量、少儿重大疾病数量、少儿特定重大疾病数量 | 少儿产品专用，成人产品此组填0或N/A |

**页面实现要求：**
- 清单区中关联字段以"关联组"形式展示，点击一个字段时同组字段高亮
- 中栏证据区提示关联字段的当前状态
- 出现依赖冲突时（如次数=1但分组=2组）系统自动标记警告

## 六、产品级复核决策

在正式进入字段级复核前，产品级先做一轮粗筛：

| 检查项 | 条件 | 提示 |
|---|---|---|
| 资料包完整性 | 是否有条款、说明书、费率表 | ⚠ 缺说明书 |
| 高风险字段 | 冲突字段数 > 0 | 🔴 存在X处冲突，建议先处理 |
| 缺失字段 | 缺失字段数 > 0 | ⬜ X个字段无法提取，请确认是否需补资料 |
| 来源覆盖 | 是否所有条款来源均正常解析 | ⚠ PDF解析质量低（置信度<0.6）|

产品级决策按钮：
- **进入字段复核**：资料包基本完整，直接审字段
- **打回补资料**：资料包缺失关键文件，返回给数据组
- **标注已知缺陷入库**：明确某字段无法从现有资料提取，允许带缺陷入库并注明原因

## 八、V1 vs V2 对比

| 维度 | V1 | V2 |
|---|---|---|
| 复核对象 | 单条字段候选 | 整产品 |
| 字段展示 | 一次只看一条 | 分组全览 + 单条聚焦 |
| 来源展示 | 只显示 winning source | 多来源并列，标注一致/冲突 |
| 依赖关系 | 无 | 关联字段组，警告提示 |
| 产品级状态 | 无 | 完整状态机 |
| 入库决策 | 字段接受即可提交 | 产品级+字段级双确认 |
| 数据来源 | clause + processed_rate | clause + brochure + processed_rate + underwriting |
| 测试策略 | 浏览器 iframe 测试（不稳定） | 接口契约测试 + 状态机覆盖测试 |
