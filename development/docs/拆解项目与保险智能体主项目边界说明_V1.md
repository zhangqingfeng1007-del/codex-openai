# 拆解项目与保险智能体主项目边界说明 V1

生成时间：2026-03-25

## 一、目的

本说明用于明确：

1. 保险产品智能拆解项目的职责边界
2. 保险智能体主项目的职责边界
3. 两者之间的共享契约和协作方式

目标不是把两个项目完全切开，而是做到：

1. 开发过程隔离
2. 标准库口径统一
3. 正式结果唯一

## 二、项目关系

两个项目的关系是：

1. 保险产品智能拆解项目：负责把原始资料包拆成标准库数据
2. 保险智能体主项目：负责消费标准库数据，对外提供推荐、解释、利益展示和问答能力

因此：

1. 拆解项目是主项目数据库建设链路的一部分
2. 拆解项目不是主项目前端/服务逻辑的一部分
3. 主项目不应直接依赖拆解过程产物

## 三、必须隔离的内容

以下内容必须在工程上隔离：

1. 拆解项目的脚本
2. 盲测数据
3. 中间 Markdown
4. blocks 结果
5. 候选结果文件
6. 评测脚本和评测结果
7. 规则试验代码

当前隔离目录：

1. `/Users/zqf-openclaw/codex-openai/development`

这类内容只属于拆解项目，不应直接进入主项目正式目录。

## 四、不能各自定义的共享内容

以下内容属于共享契约，不能由两个项目各自维护一套：

1. 标准项定义
2. 标准库字段映射
3. 枚举值口径
4. 入库输出结构
5. 数据来源追溯要求
6. 统一校验规则

如果这些口径不统一，会出现：

1. 拆解项目产出一套结果
2. 智能体主项目消费另一套口径
3. 同一产品在数据库和前台出现不同结论

## 五、拆解项目职责

拆解项目负责：

1. 接收原始资料包
2. 文档解析
3. 规则抽取
4. 大模型候选抽取
5. 候选结果规范化
6. 来源证据保留
7. 人工复核支持
8. 统一入库前的数据准备

拆解项目不负责：

1. 用户侧推荐逻辑
2. 产品评分逻辑
3. 对话问答逻辑
4. 前台展示策略

## 六、保险智能体主项目职责

主项目负责：

1. 消费标准库正式数据
2. 执行产品筛选与推荐
3. 执行评分与解释
4. 面向用户进行结果展示
5. 控制对外业务链路

主项目不应直接依赖：

1. Markdown 中间文件
2. blocks
3. 候选结果 JSON
4. 盲测输出
5. 临时评测脚本

## 七、正式交付边界

拆解项目交付给主项目的，不应是过程产物，而应是：

1. 标准库正式数据
2. 或统一入库结构
3. 或标准化 JSON Schema

不能直接作为主项目输入的内容：

1. `md_cache`
2. `blocks`
3. `blind_test_results`
4. `tier_a_rule_candidates`

这些内容只用于拆解项目内部开发、评测和复核。

## 八、统一原则

两个项目共同遵守以下原则：

1. 所有重要信息最终都要入标准库
2. 所有入库内容都必须有来源
3. LLM 只能给候选，不能直接形成正式业务结论
4. 正式结果只能通过统一校验和统一入库链路产生
5. 不允许多个入口产生不同正式结论

## 九、当前建议的协作方式

建议按“三层结构”协作：

### 9.1 开发层

拆解项目独立开发，放在 `development/` 下。

### 9.2 契约层

共享：

1. 字段映射
2. 枚举字典
3. 标准库目标结构
4. 入库规则

### 9.3 消费层

智能体主项目只消费正式标准库，不消费拆解中间产物。
