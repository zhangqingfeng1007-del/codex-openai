# 测试边界与答案隔离原则 V1

**制定日期：** 2026-03-26
**适用范围：** AIX 保险产品拆解项目全部评测任务
**核心原则：** 标准库可用，目标产品答案库不可用。

---

## 一、快速判断口诀

> **"这份数据，是不是这道题的标准答案？"**
> 是 → 拆解能力评测中不能用
> 否（通用规则 / 标准库 / 原始文件） → 可以用

---

## 二、数据源三类划分

### ✅ 可用：原始资料包 + 标准库

| 数据源 | 类型 | 说明 |
|---|---|---|
| 条款 PDF / MD | 原始资料 | 主要抽取来源 |
| 产品说明书 | 原始资料 | 辅助来源 |
| 费率表（raw_rate / processed_rate） | 原始资料 | 辅助来源 |
| 核保文件 | 原始资料 | 投保年龄等字段 |
| `cmb_coverage` | 标准库 | 标准项定义（字段名、ID） |
| `cmb_coverage_standard` | 标准库 | 标准值集合 |
| `coverage_standard_values_v1.json` | 标准库 | 标准值参考库本地导出 |
| `disease_library_v1.json` | 标准库 | 病种标准名称库（跨产品） |
| `coverage_whitelist_v1.json` | 规则库 | 字段白名单与评测口径配置 |
| `normalize_value.py` | 规则库 | 格式归一规则 |
| 枚举 / 映射规则 | 规则库 | 业务规则文档中的判定逻辑 |

### ❌ 不可用：该产品的已拆解结果值

以下数据等价于直接看答案，**拆解能力评测中严禁使用**：

| 数据源 | 原因 |
|---|---|
| `cmb_product_coverage` 中该产品的 `standard_content` | gold 评测同源，完全循环 |
| `product_coverage_db_v1.json` 中该产品的字段值 | 同上，结果镜像 |
| `product_disease_counts_v1.json` 中该产品的计数值 | 直接给出重疾数量答案 |
| 任何等价的产品级导出镜像 | 只要内容是该产品已拆解的结果值，均不可用 |

#### 两个本地镜像文件的具体含义

1. `product_coverage_db_v1.json`
   - 含义：按 `db_product_id` 保存该产品各标准项的已入库标准值镜像。
   - 典型内容：`保险期间`、`交费期间`、`等待期`、`重疾赔付次数` 等字段的最终 `standard_content`。
   - 结论：这是**产品字段答案镜像**，可用于联调，不可用于该产品拆解评测。

2. `product_disease_counts_v1.json`
   - 含义：按 `db_product_id` 保存该产品病种数量的已统计结果镜像。
   - 典型内容：`重大疾病 = 120`、`少儿重大疾病 = 10`。
   - 结论：这是**产品病种数量答案镜像**，不可直接作为该产品 `重疾数量` 的拆解来源。

### ⚠️ 可用但需说明：中间识别结果

**前提：该中间结果必须由原始资料重新生成，不是直接从结果库导出。**

典型合法案例：
```
条款附注 → 识别病种列表（disease_extraction_result）
  → 与 disease_library 匹配（matched_disease_list）
  → 统计数量（disease_count_summary）
  → 输出重疾数量候选值
```

- `disease_extraction_result`：合法（从原始文件生成）
- `product_disease_counts_v1.json`：**不合法**（从结果库直接导出，是答案）

关键区别：**推导出的中间结果 ≠ 直接读取的最终结果。**

## 三、能力三层定义

评测时必须明确当前验证的是哪一层，不同层允许使用的数据源不同。

### Layer 1：原始拆解能力（Extraction）

**定义：** 系统从原始文件（条款/说明书/费率表）中识别字段值的能力。

**允许来源：** 原始资料包 + 标准库
**禁止来源：** 该产品任何已结构化结果值
**评测脚本：** `eval_tier_a.py`、`eval_blind_test_v1.py`
**典型指标：** 命中率、mismatch 率、miss 率

### Layer 2：规则判定 / 标准化能力（Normalization & Rule）

**定义：** 系统把抽取的原始值转化为标准格式的能力，包括格式归一、来源优先级判定、依赖推断等。

**允许来源：** Layer 1 输出 + 标准库 + 规则库
**禁止来源：** 该产品的最终结果值
**典型能力：** `normalize_value.py`、`infer_frequency_from_pay_period()`、`standard_value_check`
**典型指标：** format_diff 率、归一后命中率提升量

### Layer 3：系统补全 / 联调能力（Completion & Integration）

**定义：** 验证 review task 构建、前端展示、任务流转等系统组件是否完整正确。

**允许来源：** 已结构化结果镜像（product_coverage_db 等均可用）
**目标：** 不是评测拆解能力，而是验证数据流通链路
**典型场景：** 验证 build_review_task.py 输出是否完整、前端页面能否正确渲染

## 四、违规示例与修正

| 做法 | 层级 | 判断 | 修正 |
|---|---|---|---|
| 交费期间走 product_coverage_db 优先 | Layer 1 评测 | ❌ 直接拿答案 | 改为条款/费率表抽取 |
| 重疾数量直接读 product_disease_counts | Layer 1 评测 | ❌ 直接拿答案 | 改为条款 → 病种识别 → 统计 |
| 用 disease_library 辅助病种名称匹配 | Layer 1/2 | ✅ 标准库参考 | 保留 |
| 用 coverage_standard_values 做格式校验 | Layer 2 | ✅ 标准库参考 | 保留 |
| 用 product_coverage_db 验证前端渲染 | Layer 3 | ✅ 联调场景 | 保留，但不计入 Layer 1/2 指标 |
