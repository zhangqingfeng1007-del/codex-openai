## 一、背景与 V1 废弃原因

当前 V1 原型（`~/codex-openai/development/review-module/`）存在两个根本问题：

**问题 1：复核对象建模错误**

V1 以"单条字段候选"为复核单元（每次只看一个 candidate）。
但真实人工复核是以**产品**为单元：

- 复核员需要同时看整个产品的字段完整性
- 需要看同一字段在多个来源（条款/说明书/费率表）的一致性
- 需要看字段间的依赖关系（如重疾赔付次数 ↔ 重疾分组）

**问题 2：架构不稳定**

- 状态存 localStorage，测试容易被污染
- 没有任务级状态机，不知道整个产品复核进度
- mock 数据和页面逻辑耦合，接真实接口就会出问题

## 二、你的任务

请你先**分析**以下三个问题，给出设计方案。不要写前端代码，只输出三份 Markdown 设计文档。

### 任务 A：产品级复核工作台设计

输出文件：`人工复核工作台_V2_产品级设计.md`

请你设计一个**产品级复核工作台**，回答以下问题：

1. **页面分区**：工作台应该有哪几个功能区？每个区负责什么？
   - 提示：至少需要标准项清单区、多来源证据区、人工裁决区
   - 标准项清单应按分组展示（基础字段/重疾责任/少儿责任等），不是一条一条线性滚动

2. **多来源展示**：同一字段在条款、说明书、费率表有不同值时，如何展示？
   - 来源优先级：条款 > 说明书 > 核保规则 > 费率表
   - 费率表投保年龄只有 0 周岁占位，不含天数，不能作为天数来源
   - 冲突情况要高亮提示，给出系统建议（不强制）

3. **字段分组**：标准项应如何分组展示？
   - 建议分组：基础规则字段（投保年龄/保险期间/交费期间等）、重疾责任字段（次数/分组/数量）、少儿责任字段
   - 冲突字段和缺失字段应有动态分组（自动聚合），优先展示

4. **产品级决策**：进入字段级复核前，产品级应先做哪些检查？
   - 资料包完整性（是否有条款/说明书/费率表）
   - 高风险字段数（冲突/缺失）
   - 操作：进入字段复核 / 打回补资料 / 标注缺陷允许入库

5. **数据契约**：V2 的 API 响应结构应包含哪些字段？
   - 特别关注：如何传递多来源候选、依赖警告、字段分组

### 任务 B：信息架构设计

输出文件：`人工复核信息架构_V2.md`

请设计信息层级结构和展示密度控制，回答：

1. 信息层级：Task → FieldGroup → CoverageItem → Source 各层包含哪些字段？
2. 清单行展示什么？详情层展示什么？默认展开/折叠规则？
3. 来源类型（clause/brochure/processed_rate/raw_rate/underwriting_rule）各自的可信度和适用范围？
4. 冲突类型如何分类？（来源值不同 vs 格式差异 vs 语义冲突 vs 依赖冲突）
5. 依赖关系有哪些？（重疾责任链 / 投保年龄-天数链 / 交费三联 / 少儿责任链）

### 任务 C：状态机设计

输出文件：`人工复核模块状态机_V2.md`

请设计两层状态机：

1. **任务级状态机**（TaskStatus）：状态定义、转移条件、转移规则
   - 至少包含：pending / in_review / review_completed / returned_for_materials / import_submitted
   - 什么条件下任务才能转为 review_completed？（所有字段必须裁决完）

2. **字段级状态机**（CoverageItemStatus）：状态定义、转移条件、转移规则
   - 至少包含：review_required / candidate_ready / accepted / modified / rejected / cannot_extract / cannot_extract_from_clause
   - 哪些状态阻断入库？哪些不阻断？

3. **入库决策规则**：什么条件下允许提交入库？rejected 字段如何处理？cannot_extract 字段如何处理？

4. **依赖触发规则**：当哪些字段变更时触发哪些联动检查？
   - 重疾赔付次数变更 → 检查分组合理性
   - 投保年龄 0-X周岁 → 提示确认天数来源
   - 交费期间含趸交 → 检查交费频率合理性

5. **前端状态管理要求**：V2 应如何存储状态？（提示：不能再用 localStorage，裁决应实时 POST）
