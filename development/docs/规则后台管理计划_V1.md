# 规则后台管理计划 V1

版本：V1

日期：2026-04-04

适用范围：保险条款智能拆解项目

## 1. 目标

当前项目已经证明：大量拆解问题本质上不是“模型理解力不够”，而是字段级规则没有被稳定管理。

因此，后续必须建设“规则后台管理能力”，让人工能够通过后台配置拆解规则，而不是每次都改代码。

后台管理能力的目标是：

1. 人工可配置字段拆解规则
2. 规则变更可追踪、可回滚
3. 机器拆解优先读取配置而不是硬编码
4. 人工审核反馈能反哺规则更新

## 2. 背景

### 2.1 当前现实约束

当前人工拆解团队仍承担繁重的生产拆解任务，无法集中投入大量时间一次性把规则文档补全。

因此，机器拆解系统必须尽快形成可投入生产辅助的能力，优先减轻人工负担。

### 2.2 当前主要问题

当前若没有后台管理，会持续出现：

- 改一个字段要改代码
- 规则变化无法快速同步
- 同一个问题反复讨论
- 现状、规则、代码三者脱节
- 人工审核发现的问题不能快速沉淀成配置

## 3. 设计原则

### 3.1 配置优先

只要属于字段级规则，不应优先写死在代码里，应优先沉淀为配置。

### 3.2 代码只承载通用能力

代码层负责：

- 读取规则配置
- 执行通用策略
- 调用解析器 / 匹配器

规则层负责：

- 哪个字段启用
- 去哪里找
- 怎么过滤
- 无证据如何处理
- 是否 direct_extract / semantic / exact

### 3.3 人工可维护

后台配置必须让熟悉业务的人可以直接维护，而不是只能由工程师改代码。

### 3.4 变更可审计

所有规则变更必须可追踪：

- 谁改了
- 改了什么
- 为什么改
- 何时生效
- 能否回滚

## 4. 后台需要支持的配置能力

### 4.1 字段启停

可配置：

- 是否纳入当前主线
- 是否仅实验启用
- 是否已挂起
- 是否后置

### 4.2 来源规则

可配置两个维度：

- `source_priority`（有序优先级）：按顺序尝试从哪类材料取值，先命中者优先
  - 例：`[条款正文, 说明书, 费率表]`
- `disallowed_materials`（禁止来源）：明确不允许从哪类材料取值
  - 例：`[销售材料, 核保文件]`

不再单独设 `preferred_materials`，由 `source_priority` 隐含表达允许范围。

### 4.3 字块过滤规则

可配置：

- 是否跳过举例
- 是否跳过目录
- 是否跳过释义
- 是否跳过营销块
- 额外跳过词

### 4.4 无证据处理

可配置：

- 置空
- 不涉及
- 挂起
- direct_extract_only

### 4.5 匹配策略

可配置：

- exact_match
- semantic_candidate_match
- direct_extract
- manual_only
- no_standard_match

### 4.6 值标准化规则

可配置结构化标准化操作（对应规范中的 `value_normalization`）：

- `strip_suffix: [str]`
- `precision: year_only | month_only | day`
- `replace: {from, to}`
- `trim: true`

此字段取代原先硬编码的特殊处理（如投保年龄忽略天月、合同名称尾部清洗）。

### 4.7 业务特殊规则

可配置无法结构化的业务判断（对应规范中的 `special_rules`）：

- 豁免无责任则置空
- 等待期允许范围补全兜底
- 附加险引用主合同时挂起

### 4.8 样本与验证信息（第二期）

以下信息第一期在 YAML 文件中静态维护，第二期再纳入后台可配置：

- 验证样本产品
- 已知风险
- 当前状态

## 5. 后台数据模型建议

后台最小应能维护一张字段规则表。

建议字段如下：

- `coverage_name`
- `coverage_id`
- `coverage_path`
- `domain`
- `enabled`
- `source_priority`（有序优先级，取代原 preferred_materials）
- `disallowed_materials`
- `section_hints`
- `block_rules`
- `extraction_strategy`
- `value_normalization`
- `evidence_strategy`
- `no_coverage_behavior`
- `no_evidence_behavior`
- `match_strategy`
- `special_rules`
- `status`
- `validation_samples`（第二期纳入后台）
- `known_risks`（第二期纳入后台）
- `rule_version`
- `updated_by`
- `updated_at`
- `change_reason`

## 6. 分期计划

### 第一期：配置文件化

目标：

- 字段规则从口头/散文文档变成结构化配置文件
- 代码按配置执行

形态：

- YAML / JSON 规则文件
- 本地版本管理

最小交付物：

1. 规则 YAML schema 定稿（字段结构与规范文档 §5 对齐）
2. 所有已上线字段（≥20个）完成配置迁移
3. `extract_responsibility_fields.py` 和 `extract_tier_a_rules.py` 读取配置而非硬编码
4. dry-run 回归通过（sample 10 + blind test 匹配数不退化）

验收标准：

- 新增/修改一个字段的来源、过滤、匹配策略，只需改 YAML 配置，不需改 Python 代码
- 配置文件可被 `json.load` / `yaml.safe_load` 正确解析，无自然语言歧义

收益：

- 多数字段规则调整无需改代码
- 当前主线可以快速稳定下来

### 第二期：后台编辑与发布

目标：

- 提供后台页面维护字段规则
- 支持规则版本、发布、回滚

最小功能：

- 编辑字段规则
- 对比变更
- 发布新版本
- 查看影响范围

### 第三期：与人工审核联动（远期）

目标：

- 人工审核中发现的问题可沉淀为规则变更建议
- 后台形成规则闭环

当前阶段人工审核量不足以支撑自动归类，第三期以手动沉淀为主：

- 人工审核发现的共性问题，手动录入规则变更
- 待审核量积累后，再考虑自动归类和建议生成

## 7. 当前实施建议

当前阶段不急于直接开发完整后台，而应先完成：

1. 字段规则结构定稿
2. 字段规则文件落地
3. 代码开始读取字段规则
4. 用当前主线字段验证配置驱动模式

也就是说，当前应先完成“后台化前的结构准备”，再进入真正的后台系统开发。

## 8. 当前结论

后续机器拆解规则必须朝以下方向演进：

- 字段级规则可配置
- 规则变更可审计
- 人工审核可反哺规则
- 规则不再依赖工程师手改代码才能生效

本文件只定义后台管理计划，不替代字段规则规范文档。
