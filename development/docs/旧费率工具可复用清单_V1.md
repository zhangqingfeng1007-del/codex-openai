# 旧费率工具可复用清单 V1

生成时间：2026-03-25
分析对象：`/Users/zqf-openclaw/Desktop/开发材料/rate.rar`

## 一、结论摘要

这套旧费率工具不能直接作为新系统组件接入，但其中沉淀了较多可复用的业务规则和标准化口径。最有价值的是：

1. `datas.py` 中的业务字典和标准化映射
2. `rate_final_result.py` 中的结果费率表结构
3. `database_format.py` 中“结果表 -> 数据库字段”的映射思路

最不建议直接复用的是 PDF 识别主流程：

1. `rate_recognition.py`
2. `rate_model.py`
3. `rate_tools.py`

原因是代码耦合高、重复多、依赖旧工程环境、可测试性差。

## 二、归档结构

`rate.rar` 内核心文件：

1. `rate/datas.py`
2. `rate/rate_tools.py`
3. `rate/rate_recognition.py`
4. `rate/rate_model.py`
5. `rate/rate_final_result.py`
6. `rate/database_format.py`

从职责上看，整体流程是：

1. PDF 费率表识别
2. 生成中转表
3. 中转表生成结果费率表
4. 结果费率表转数据库格式

## 三、直接复用

### 3.1 `datas.py` 的业务字典

建议直接复用或迁移重组的内容：

1. 公司名 / 公司编号映射
2. `保险期间` 规范化字典
3. `交费期间` / `趸交` / `交至XX周岁` 映射
4. `性别`、`社保`、`吸烟状态`、`保费状态` 枚举
5. 结果费率表表头定义
6. 数据库字段映射定义

原因：

1. 这些内容本质是业务口径，不依赖旧代码结构
2. 与当前“标准库入库”目标直接相关
3. 可沉淀成新的配置文件或字典模块

### 3.2 结果费率表字段定义

`datas.final_df_head` / `datas.final_df_head_3` 已经定义了一套较稳定的结果表结构，建议直接作为新系统费率标准化结果的参考基线。

典型字段包括：

1. 公司编号 / 公司 / 产品编号 / 产品
2. 投保年龄
3. 缴费期间
4. 保险期间
5. 性别
6. 固定保额 / 份数
7. 交费方式
8. 保费状态
9. 费率

## 四、重写复用

### 4.1 `database_format.py`

这部分逻辑值得重写复用，不建议原样继承。

可借鉴点：

1. 结果表字段到数据库字段的映射思路
2. 特殊值字典映射
3. 数值字段 + 类型字段双列输出方式
4. 默认值补齐逻辑

不建议直接复用原因：

1. 配置和代码混在一起
2. 异常处理较弱
3. 可读性一般
4. 命名和风格不适合新项目

建议新做法：

1. 将字段映射定义迁移到配置层
2. 将转换逻辑做成独立、可测试模块
3. 输出统一标准库入库格式

### 4.2 `rate_final_result.py`

这部分也建议“保留思路，重写实现”。

可借鉴点：

1. 中转表按列展开成结果表的方式
2. 固定保额 / 保费 / 份数处理逻辑
3. 结果表按“产品维度 + 年龄维度 + 交费维度”展开

不建议直接复用原因：

1. 强依赖中转表格式
2. 函数体过长，逻辑集中度差
3. 针对旧业务场景有较多隐式假设

## 五、仅参考，不直接复用

### 5.1 `rate_recognition.py`

这是旧工具中 PDF 表格识别的主流程，但不建议直接接入。

问题：

1. 强依赖 `pdfplumber`
2. 行列清洗逻辑堆叠严重
3. 变量状态很多，可维护性差
4. 与 Excel 输出、Django 设置、业务规则耦合过深

可参考点：

1. 表格外信息回填思路
   例如从页文本中补 `保险期间`、`性别`、`交费方式`
2. 表格异常修复思路
   例如：
   - 删除全空行
   - 删除全空列
   - 年龄区间展开
   - 一行双年龄切分

### 5.2 `rate_model.py`

该文件与 `rate_recognition.py` 有明显重复，像是旧版本重构未完成的残留。

结论：

1. 不建议复用
2. 仅在需要追溯历史逻辑时参考

### 5.3 `rate_tools.py`

这个文件主要是 PDF 字符级读取和一些辅助清洗方法。

可参考点：

1. `read_pdf`
2. 行文本拼接
3. 年龄区间展开

不建议直接复用原因：

1. 与当前 `doc_to_md` 路线不一致
2. 依赖底层字符坐标拼接，调试成本高
3. 代码风格与新项目不兼容

## 六、对当前项目的实际价值

对我们当前最有价值的，不是“直接拿来跑费率解析”，而是以下三层资产：

1. 历史业务字典
2. 历史结果费率表结构
3. 历史数据库字段映射

这三层资产可以帮助我们：

1. 不从零定义费率标准库
2. 保持和历史人工拆解结果一致
3. 缩短新费率层建设时间

## 七、建议的接入策略

### 7.1 第一阶段

先不碰旧 PDF 识别主流程，只做：

1. 提取 `datas.py` 的业务字典
2. 对齐当前处理后费率表结构
3. 确认标准库目标字段

### 7.2 第二阶段

重写“处理后费率表 -> 标准库格式”转换层。

优先目标：

1. 投保年龄
2. 交费期间
3. 交费频率
4. 保险期间
5. 费率矩阵结构

### 7.3 第三阶段

再决定是否重做“原始 PDF/Excel 费率表 -> 中间结构”解析层。

这一层建议新建，不建议直接继承旧实现。

## 八、最终判断

这套旧费率工具的价值主要在“业务规则”和“结果结构”，不在“旧代码实现本身”。

适合直接拿来用的：

1. 字典
2. 枚举
3. 表头
4. 数据库字段映射

适合重写后复用的：

1. 结果表生成逻辑
2. 数据库转换逻辑

只适合参考、不适合继承的：

1. PDF 识别主流程
2. 字符级拼接和表格修复主逻辑
