# 标准入库输出结构 V1

生成时间：2026-03-25

## 一、目的

本结构定义拆解项目向标准库交付的正式输出格式。目标是让拆解项目和保险智能体主项目通过统一结构交接，而不是通过中间产物交接。

## 二、总体原则

1. 正式输出必须贴近标准库表结构
2. 每条记录必须带来源信息
3. 候选结果与正式结果要区分
4. 主项目只消费正式结果

## 三、条款类输出

目标表：

1. `cmb_product_coverage`

建议标准输出结构：

```json
{
  "product_id": "1010003851",
  "coverage_id": 4,
  "coverage_name": "保险期间",
  "standard_content": "10年",
  "source_type": "clause",
  "source_file": "700-招商信诺真惠保重大疾病保险（互联网专属）-条款.pdf",
  "evidence": {
    "page": 3,
    "block_id": "1010003851_p3_b24",
    "text": "保险期间为10年。"
  },
  "extract_method": "rule: insurance_period_pattern",
  "review_status": "review_passed"
}
```

## 四、费率类输出

目标表：

1. `cmb_product_rate`

当前核心输出字段：

1. `product_id`
2. `amount`
3. `rate`
4. `age`
5. `gender`
6. `payment_years`
7. `insurance_period`
8. `pay_type`

```json
  "rate": 214.5,
  "age": 0,
  "gender": 11,

  "review_status": "review_passed"

```

## 六、候选结果与正式结果区分

候选结果：

1. 可保存在中间表或中间 JSON
2. 允许存在 `review_required`
3. 不构成业务正式真相

正式结果：

1. 必须完成统一校验
2. 必须有来源
3. 必须通过复核或自动通过规则
4. 才能进入标准库

## 七、字段级必带元数据

所有正式输出建议统一携带：

1. `source_type`
2. `source_file`
3. `extract_method`
4. `review_status`

条款类字段还应带：

1. `evidence.page`
2. `evidence.block_id`
3. `evidence.text`

## 八、禁止事项

1. 不允许主项目直接读取 `md_cache`
2. 不允许主项目直接读取 `blocks`
3. 不允许主项目直接读取候选 JSON 作为正式结果
4. 不允许无来源数据进入标准库

## 九、当前实现顺序

1. 先稳定条款类输出结构
2. 再稳定费率类输出结构
3. 再扩病种和其他资料源
