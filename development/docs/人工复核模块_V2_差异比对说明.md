# 人工复核模块 V2 差异比对说明

**用途：** 对比 Codex 版与 Claude 草稿，供项目负责人裁决  
**对比对象：**
1. 人工复核工作台 V2
2. 人工复核信息架构 V2
3. 人工复核模块状态机 V2

---

## 一、工作台设计差异

### 一致点

1. 都认为 V1 的根本问题是“以字段为主对象”
2. 都把 V2 提升为产品级复核工作台
3. 都采用五区结构
4. 都要求多来源并排、依赖关系显式化

### 差异点

1. Claude 草稿更强调页面布局和视觉分区
2. Codex 版更强调任务对象、字段分组、依赖组和接口契约

### Codex 方案更优的点

1. 明确把 `ReviewTask(Product)` 定义为页面主对象
2. 把“依赖组”单独建模，不只是描述性说明
3. 接口 JSON 更适合直接指导后端实现

### Claude 草稿更稳的点

1. 页面五区描述更完整
2. 示例更接近真实操作场景
3. 对复核员视角更友好

### 建议最终采纳方案

1. 保留 Claude 的五区布局表达
2. 吸收 Codex 的 `dependency_group` 建模和任务对象表达

---

## 二、信息架构差异

### 一致点

1. 都认为信息层级必须超过 `Task -> Item`
2. 都支持多来源候选
3. 都要求冲突分类和来源可信度区分

### 差异点

1. Claude 草稿更强调字段分组和动态分组的业务视图
2. Codex 版更强调 `Task -> Product -> Package -> Group -> Item -> Source` 的工程层级

### Codex 方案更优的点

1. 分层更接近后端对象模型
2. 更容易映射到接口和前端状态树
3. 清楚区分 `ProductSnapshot` 和 `DocumentPackage`

### Claude 草稿更稳的点

1. 对冲突类型、依赖关系和产品级信息的业务表达更丰富
2. 对复核页默认展开/折叠的说明更细

### 建议最终采纳方案

1. 以 Codex 的对象层级为骨架
2. 采用 Claude 的动态分组和默认展示策略

---

## 三、状态机差异

### 一致点

1. 都采用任务级 + 字段级双状态机
2. 都反对 V1 的 localStorage 状态
3. 都要求裁决后端持久化
4. 都要求依赖字段变更触发联动检查

### 差异点

1. Claude 草稿的状态机更完整，覆盖到 `import_failed`、`archived`
2. Codex 版更强调 `ImportStatus` 和事件日志

### Codex 方案更优的点

1. 更明确指出状态机必须配套事件流
2. 更适合后续审计和可追溯要求
3. 对前端状态管理边界更清楚

### Claude 草稿更稳的点

1. 状态定义更细
2. 转移图更完整
3. 入库决策和阻断条件更接近可执行规范

### 建议最终采纳方案

1. 状态枚举和转移规则优先采用 Claude 草稿
2. 事件日志和前端状态边界吸收 Codex 版

---

## 四、总体裁决建议

### 建议保留 Claude 为主稿的部分

1. 工作台布局
2. 动态分组描述
3. 任务级/字段级状态枚举与转移规则

### 建议采纳 Codex 的补强部分

1. `ReviewTask(Product)` 主对象定义
2. `dependency_group` 联动审核组
3. `ProductSnapshot / DocumentPackage` 分层
4. 事件日志模型
5. 前端只保留视图状态，裁决实时 POST

### 最终建议

最终正式版 V2 建议采用：

1. **Claude 草稿为主结构**
2. **Codex 方案补强工程模型和事件日志**

原因：
1. Claude 草稿在业务视角和页面组织上更成熟
2. Codex 方案在工程可实现性上更干净
3. 两者不是互斥，更适合合并

---

## 五、项目负责人裁决建议

推荐你按下面方式快速裁决：

1. 页面布局与信息组织：以 Claude 为主
2. 对象模型与接口骨架：以 Codex 为主
3. 状态机枚举：以 Claude 为主
4. 事件日志与前端状态边界：采纳 Codex

这样合并后，就可以进入下一轮前端重写，而不是继续修 V1。
