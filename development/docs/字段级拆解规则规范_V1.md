# 字段级拆解规则规范 V1

版本：V1

日期：2026-04-04

适用范围：保险条款智能拆解项目

当前主线：重疾主险

## 1. 文档定位

本文件只定义“字段级拆解规则”的规范，不记录当前项目完成度、基线数据、实验结果或临时任务安排。

本文件回答的是：

- 一个标准项应去哪里找结论
- 哪些材料可用，哪些材料不可用
- 哪些字块可取，哪些字块必须跳过
- 无证据时如何处理
- 该字段是否参与标准值匹配
- 该字段适用 exact、semantic、direct extract 中哪一种

本文件不回答的是：

- 当前哪些字段已经做完
- 当前盲测基线是多少
- 当前哪个样本命中过
- 当前哪个实验版本效果如何

这些内容应写入独立的“项目现状文档”。

## 2. 建设原则

### 2.1 规则优先于代码

每次处理一个标准项时，必须先查字段规则，再决定是否开发和如何开发。

执行顺序固定为：

1. 查询字段级拆解规则
2. 判断该字段是否属于当前主线
3. 判断应从哪类材料找结论
4. 判断无证据时如何处理
5. 再执行提取逻辑
6. 再执行标准值匹配或人工审核

禁止倒置为：

- 先跑提取器
- 再靠人解释字段逻辑

### 2.2 人工规则处于持续建设状态

过去没有完整的人工作业规则库。当前人工拆解规则仍在持续补充中，短期内不会一次性完整。

因此：

- 文件中已明确写出的规则，当前优先采用
- 文件中尚未写出的字段或细则，不代表不能开发
- 对规则未完备、但样本充分、标准值清晰、收益明确的字段，允许基于标准值词典和样本先行实现最小可用版本
- 若现有机器拆解逻辑明显优于当前人工规则，应主动提出，由人工确认后再反哺规则

### 2.3 当前目标是尽快形成生产辅助能力

当前项目目标不是等待人工规则文档完整后再开发，而是尽快形成可投入生产辅助的机器拆解能力，以减轻人工拆解工作量。

因此，是否推进一个字段，优先看：

1. 是否能显著减轻人工负担
2. 当前样本是否足以验证
3. 风险是否可控
4. 是否适合先做 v0 / v1

而不是只看：

- 规则文档是否已经写完

### 2.4 规则表达必须简洁可执行

人工语言允许存在解释性废话，但机器拆解规则必须压缩为最小可执行逻辑。

每条规则只保留：

- 去哪里找
- 什么能取
- 什么不能取
- 无证据怎么办
- 如何匹配

禁止把大段人工说明直接作为机器规则。

### 2.5 规则最终要支持后台配置

字段规则的设计必须为后续后台配置管理做准备，不应写成只能人看懂的散文。

因此，每个字段规则必须能被结构化存储，并支持后台配置、版本记录、审计与回滚。

## 3. 规则优先级

字段处理优先级从高到低：

1. PM / 人工明确口径
2. 字段级拆解规则
3. 核心系统拆解规则文档
4. 当前代码实现
5. 历史临时结论

若代码实现与字段规则冲突，以字段规则为准。

## 4. 当前主线与排除范围

### 4.1 当前主线

当前主线只处理：

- 重疾主险结构化字段

### 4.2 当前不纳入主线

以下类别不纳入当前主线：

- 附加险单文件问题
- 医疗险续保字段
- 招行专用字段
- 规则未完备且材料不足的投核保规则字段
- 暂缓的长文本复杂说明字段

### 4.3 后置字段

以下字段后置，待补材料后再开：

- 保额要求
- 保费要求

原因：

- 主要应从产品说明书、投保规则、核保文件中找结论
- 当前材料不全

## 5. 字段规则结构

每个字段规则至少包含以下部分。

### 5.1 基础标识

- `coverage_name`
- `coverage_id`
- `coverage_path`
- `domain`
- `enabled`

### 5.2 来源规则

- `source_priority`：有序优先级列表，按顺序尝试取值，先命中者优先。隐含表达允许范围，不再单独设 `preferred_materials`。
- `disallowed_materials`：明确禁止的材料类型。
- `section_hints`：提示优先搜索的条款章节关键词。

### 5.3 字块过滤规则

- `must_be_matchable`
- `skip_example`
- `skip_toc`
- `skip_definition`
- `skip_marketing`
- `extra_skip_patterns`

### 5.4 提取规则

- `extraction_strategy`
- `candidate_format`
- `special_rules`

`extraction_strategy` 允许值：

- `rule_extract`：基于正则/关键词的规则提取（已验证字段）
- `rule_extract_v0`：基于标准值词典反推的最小实现（实验字段）
- `direct_extract`：产品唯一值，不参与标准值匹配
- `llm_extract`：需要 LLM 辅助的复杂提取（预留，当前未启用）

### 5.4.1 值标准化规则

- `value_normalization`

结构化描述提取值的清洗逻辑，不写自然语言，直接支持机器执行。

允许的标准化操作类型：

- `strip_suffix: [str]`：删除尾部指定字符串
- `precision: year_only | month_only | day`：精度控制
- `replace: {from: str, to: str}`：字符替换
- `trim: true`：首尾空格/标点清理

示例：

```yaml
value_normalization:
  - strip_suffix: ["条款", "利益", "利益条款"]
  - trim: true
```

**硬约束：`special_rules` 与 `value_normalization` 的边界**

- 所有字符串清洗（去后缀、去前缀、替换）、精度统一（年/月/天）、首尾修整，必须写入 `value_normalization`，禁止写入 `special_rules`。
- `special_rules` 只允许承载无法结构化的业务判断逻辑，例如"附加险引用主合同时挂起""有同组证据但无法确定组数时留空待人工"。
- 评审规则定义时，如果 `special_rules` 中出现可被 `value_normalization` 操作类型覆盖的内容，必须迁移。

### 5.5 证据规则

- `evidence_strategy`
- `preferred_block_scope`
- `max_chars`

### 5.6 无覆盖处理

- `no_coverage_behavior`

当产品根本不存在该险种/功能时的处理方式。

允许值：

- `not_applicable`：输出"不涉及"
- `empty`：置空
- `skip`：不输出该字段

**判定原则：何时用 `skip`，何时用 `not_applicable`**

- `skip`：该字段是所有产品共有的基础属性（如保险期间、交费期间、宽限期、犹豫期、投保年龄、合同名称）。产品必然存在该属性，不存在"不涉及"的语义，找不到时应置空或挂起，不应输出"不涉及"。
- `not_applicable`：该字段是部分产品才有的险种/功能子项（如轻症分组、中症赔付次数、轻症赔付时间间隔、保障说明）。当产品根本不含该险种时，"不涉及"是有业务含义的正确值，应主动输出。

简记：**基础属性用 `skip`，险种子项用 `not_applicable`。**

### 5.7 无证据处理

- `no_evidence_behavior`

当该险种/功能存在，但条款中未找到该字段值时的处理方式。

允许值：

- `empty`：置空（待人工补充）
- `default_value: <值>`：输出指定默认值（如"无"）
- `suspended`：挂起，标记待补材料

### 5.8 匹配规则

- `match_strategy`

允许值：

- `exact_match`：字符级精确匹配标准值
- `semantic_candidate_match`：语义范围补全匹配（仅限白名单字段兜底）
- `direct_extract`：不参与标准值匹配，直接保留提取值
- `manual_only`：仅人工审核，机器不匹配
- `no_standard_match`：该字段无标准值体系

### 5.9 验证信息

- `validation_samples`
- `known_risks`
- `status`

## 6. 统一规则

### 6.1 举例类内容统一忽略

以下内容统一视为噪音，不进入主提取链：

- 举例
- 示例
- 案例
- 利益演示

### 6.2 合同名称（条款名称）

合同名称属于产品唯一值字段，不参与标准值匹配。

处理方式：

- 走 `direct_extract`
- 保留提取值和来源证据
- 不纳入 exact / semantic / unmatched 统计

提取后统一清洗尾部：

- `条款`
- `利益`
- `利益条款`

### 6.3 投保年龄

投保年龄统一只到年岁，不到天/月。

例如：

- `0（28天）-55周岁`
- `0（30天）-55周岁`
- `0（1个月）-55周岁`

统一视作：

- `0-55周岁`

后续与阿里云库验证时，也统一忽略天/月精度差异。

### 6.4 豁免责任字段

在“豁免责任”一级项下：

- 如果没有检索到真实责任，则置空
- 不输出 `不涉及`
- 不输出默认否定值

### 6.5 医疗险续保字段排除

以下字段按医疗险字段处理，不纳入当前重疾主线：

- 是否保证续保
- 保证续保期限
- 最大续保年龄
- 停售是否可以续保

### 6.6 招行字段排除

所有招行专用字段当前不进入拆解范围。

## 8. 新字段接入流程

后续任何新字段进入开发前，必须先完成：

1. 查询核心系统拆解规则文档
2. 判断该字段是否属于当前主线
3. 判断应从哪类材料找结论
4. 在字段规则表中新增该字段
5. 再允许进入开发

未完成以上步骤，不得直接开发。
