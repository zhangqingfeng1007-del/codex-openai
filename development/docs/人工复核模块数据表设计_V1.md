# 人工复核模块数据表设计 V1

生成时间：2026-03-26

## 一、模块定位

人工复核模块是独立模块，有自己的数据表。

- 不直接读或写 `cmb_product_coverage`（正式标准库）
- 只读拆解任务中间表，写复核结果中间表
- 入库由标准入库模块统一负责，通过接口读复核结果表

---

## 二、表结构

### 2.1 `clause_review_task`

记录一次复核任务的整体状态。

```sql
CREATE TABLE clause_review_task (
  task_id           VARCHAR(64)   NOT NULL PRIMARY KEY,
  product_id        VARCHAR(64)   NOT NULL,
  product_name      VARCHAR(255)  NOT NULL,
  document_package_id VARCHAR(64) DEFAULT NULL,
  task_status       VARCHAR(32)   NOT NULL DEFAULT 'review_required',
  -- 枚举：review_required / review_in_progress / review_completed / import_pending / imported
  rule_version      VARCHAR(32)   DEFAULT NULL,
  model_version     VARCHAR(32)   DEFAULT NULL,
  total_items       INT           NOT NULL DEFAULT 0,
  reviewed_items    INT           NOT NULL DEFAULT 0,
  created_at        DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at        DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

说明：
- `task_status` 随审核进度更新，`review_completed` 后方可触发入库
- `total_items` / `reviewed_items` 由后端计算维护，不由前端传入

---

### 2.2 `clause_review_candidate`

记录每个字段的候选值和复核结果。是复核模块的核心表。

```sql
CREATE TABLE clause_review_candidate (
  candidate_id        VARCHAR(64)   NOT NULL PRIMARY KEY,
  task_id             VARCHAR(64)   NOT NULL,
  product_id          VARCHAR(64)   NOT NULL,

  -- 字段映射
  coverage_id         INT           DEFAULT NULL,
  coverage_name       VARCHAR(128)  NOT NULL,

  -- 三值结构
  raw_value           TEXT          DEFAULT NULL,   -- 从原始来源提取，不做规范化
  normalized_value    TEXT          DEFAULT NULL,   -- 规则/模型规范化后的候选值
  final_value         TEXT          DEFAULT NULL,   -- 人工确认的最终入库值

  -- 提取信息
  extract_method      VARCHAR(128)  DEFAULT NULL,
  confidence          DECIMAL(4,2)  DEFAULT NULL,
  status              VARCHAR(32)   NOT NULL DEFAULT 'review_required',
  -- 枚举：review_required / accepted / rejected / cannot_extract / cannot_extract_from_clause

  -- 来源信息
  source_type         VARCHAR(32)   DEFAULT NULL,
  -- 枚举：clause / raw_rate / processed_rate / product_brochure / multi_source
  source_file         VARCHAR(512)  DEFAULT NULL,
  source_document_id  VARCHAR(64)   DEFAULT NULL,
  source_page         INT           DEFAULT NULL,
  source_raw_value    TEXT          DEFAULT NULL,

  -- 证据与链路（JSON存储，不做列拆分，V1阶段）
  evidence_json       JSON          DEFAULT NULL,
  -- 结构：[{"page": 7, "block_id": "xxx", "text": "..."}]
  logic_trace_json    JSON          DEFAULT NULL,
  -- 结构：{"priority_trace": [...], "normalization_trace": [...]}

  -- 复核结果
  action              VARCHAR(16)   DEFAULT NULL,
  -- 枚举：accept / modify / reject / hold
  reason_code         VARCHAR(64)   DEFAULT NULL,
  review_comment      TEXT          DEFAULT NULL,
  reviewer            VARCHAR(64)   DEFAULT NULL,
  review_time         DATETIME      DEFAULT NULL,

  created_at          DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at          DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  INDEX idx_task_id (task_id),
  INDEX idx_product_id (product_id),
  INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

说明：
- `evidence_json` 和 `logic_trace_json` V1 阶段用 JSON 列存储，不展开为子表；V2 再根据查询需求决定是否拆分
- `final_value` 默认为 NULL，接受时由后端从 `normalized_value` 复制，修改时由审核员填写
- `action` 写入后不能由前端直接改 `status`，需后端根据 action 计算 status

---

### 2.3 `clause_review_log`

记录每次操作日志，支持撤回和追溯。

```sql
CREATE TABLE clause_review_log (
  log_id          BIGINT        NOT NULL AUTO_INCREMENT PRIMARY KEY,
  candidate_id    VARCHAR(64)   NOT NULL,
  task_id         VARCHAR(64)   NOT NULL,
  operator        VARCHAR(64)   NOT NULL,
  action          VARCHAR(16)   NOT NULL,   -- accept / modify / reject / hold / revert
  old_status      VARCHAR(32)   DEFAULT NULL,
  new_status      VARCHAR(32)   DEFAULT NULL,
  old_final_value TEXT          DEFAULT NULL,
  new_final_value TEXT          DEFAULT NULL,
  reason_code     VARCHAR(64)   DEFAULT NULL,
  comment         TEXT          DEFAULT NULL,
  operated_at     DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,

  INDEX idx_candidate_id (candidate_id),
  INDEX idx_task_id (task_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

说明：
- 每次前端提交 `action` 时，后端写一条 log，再更新 candidate 表
- 撤回（revert）时记录 revert 日志，回填 old_final_value
- V1 不需要前端展示操作历史，但后端必须写，方便事后审计

## 三、字段状态流转

```
clause_review_candidate.status:

  review_required
      ├── accept → accepted（final_value = normalized_value）
      ├── modify → accepted（final_value = reviewer input）
      └── reject → rejected（final_value = NULL）

  accepted / rejected
      └── revert → review_required（撤回，清空 action 和 final_value）

  cannot_extract / cannot_extract_from_clause
      → 不可操作，页面直接标注跳过

clause_review_task.task_status:

  review_required
      → review_in_progress（有任意字段被操作后）
      → review_completed（所有 review_required 字段均已处理）
      → import_pending（触发入库请求后）
      → imported（入库模块确认写入正式库后）
```

---

## 四、与拆解中间表的关系

```
clause_parse_task（拆解任务）
    ↓ 拆解完成后，由任务调度写入
clause_review_task（复核任务，1:1 对应拆解任务）
    ↓ 数据由 clause_parse_candidate 同步
clause_review_candidate（复核候选项）
    ↓ 人工确认后
clause_review_log（操作日志）
    ↓ 复核完成，入库模块读取
cmb_product_coverage（正式标准库，由入库模块写入）
```

说明：
- 复核模块不直接持有 `clause_parse_block` 数据，block 内容通过 `evidence_json` 快照存储
- 入库模块只读 `clause_review_candidate` 中 `status = accepted` 的记录

## 五、reason_code 枚举表

建议在代码中维护为常量，不建立独立 DB 表：

| code | 含义 |
|------|------|
| `source_file_wrong` | 来源文件错误 |
| `source_missing_need_manual_fix` | 来源缺失，需人工补充 |
| `md_parse_wrong` | PDF→MD 转换有误 |
| `block_split_wrong` | 切块错误 |
| `rule_extract_wrong` | 规则提取错误 |
| `llm_mapping_wrong` | 模型映射错误 |
| `normalization_wrong` | 规范化错误 |
| `multi_source_priority_wrong` | 多来源优先级判断错误 |
| `coverage_schema_missing` | 标准项定义不完整 |
| `cannot_determine` | 无法判断 |

## 六、V1 不做的内容

| 功能 | 原因 |
|------|------|
| block 级子表 | V1 evidence_json 快照够用，V2 再拆 |
| 操作历史前端展示 | 后端写日志，但 V1 页面不需要展示 |
| 多人协作锁定 | V1 单人复核场景，不做并发控制 |
| 入库结果回写 | 由入库模块自行更新 task_status→imported |
