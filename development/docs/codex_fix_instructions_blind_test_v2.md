## 二、投保年龄修复

### 2.1 结论

**域知识来源优先级（已确认）：**

| 优先级 | 来源 | 说明 |
|---|---|---|
| 1 | 条款 | 主要来源，一般在"投保须知"或"保险责任"部分 |
| 2 | 产品说明书（brochure） | 条款没写清楚时在这里找 |
| 3 | 核保规则文件 | 两者都没有才看这里 |
| 4 | 费率表 | **基本不写"出生满XX天"**，只写 0 周岁占位。**例外：新华保险**会在费率表中明确标注 |

**公司惯性规律**：同一家公司不同产品对投保年龄说明位置有强烈惯性，工程化时应按公司分类处理。

当前 6 款失败的根因：

1. 费率表只给出 `0-X周岁`（年龄列 distinct 值，不含天数说明）
2. 条款中的 `出生满28天` 与年龄数字在不同 block，`extract_age()` 单 block regex 无法跨 block 捕捉
3. `merge_candidates` 中 rate-first 策略让费率侧 `0-X周岁` 直接胜出
4. 结果：gold `0（28天）-X周岁`，candidate `0-X周岁`

**禁止方案**：不能用"无来源域知识兜底（统一补28天）"方案：
- 产品 `1530003621`（中信保诚臻享惠康B款）gold 为 `0-65周岁`（无天数），盲目补 28天 会把它从 hit 变 mismatch
- 修复**必须有来源文本作为依据**，无来源不补值

### 2.2 修改位置

文件：`run_blind_test_v1.py`，函数 `run_product()`

在 `merged = merge_candidates(clause_candidates, rate_candidates)` 之后，`items = [...]` 之前，插入以下跨 block 补充逻辑：

```python
# 投保年龄 跨block 补充：若当前值为 "0-X周岁"，从全文中找 "出生满N天" 并升级
age_entry = merged.get("投保年龄")
if age_entry:
    age_val = normalize_value(age_entry.get("value", ""))
    if re.match(r"^0-\d+周岁$", age_val):
        all_text = "".join(normalize_value(b["text"]) for b in blocks)
        m_days = re.search(r"出生满(\d+)[天日]", all_text)
        if m_days:
            days = m_days.group(1)
            max_age_m = re.search(r"^0-(\d+周岁)$", age_val)
            if max_age_m:
                age_entry["value"] = f"0（{days}天）-{max_age_m.group(1)}"
                age_entry.setdefault("evidence", {})
                age_entry["evidence"]["summary"] = (
                    age_entry["evidence"].get("summary", "")
                    + f"；条款跨块补充：出生满{days}天"
                )
                age_entry["extract_method"] = (
                    (age_entry.get("extract_method") or "")
                    + "+clause_cross_block_days"
                )
```

要求：

1. `source_type` 不变（保持 `processed_rate` 或原来源，不伪造）
2. 只在找到 `出生满N天` 时才修改 value；找不到则保持原值（保护 1530003621 等无天数要求的产品）
3. 不影响已有 `0（N天）-X周岁` 的条款命中产品（match `^0-\d+周岁$` 不会命中含"天"的值）

### 2.3 验收预期

`投保年龄` 命中率：

- 当前：`40%`（4/10）
- 目标：`100%`（10/10）
- 保护检查：1530003621 不能从 hit 变 mismatch

## 三、交费频率修复

### 3.1 问题根因

当前存在两类问题：

#### A. 条款误提取（可修）

产品：

1. `1010003676`
2. `1010003677`
3. `1310003720`

根因：

条款中的以下文本属于**定义性脚注**，不是产品实际支持的交费频率声明：

- `保险费约定支付日：为本合同生效日之后每月、每季、每半年或者每年...`
- `指保险合同生效日在每月或每年...`

当前规则把这类定义文本误识别为产品支持 `月交/季交/半年交/年交`。

#### B. 数据源缺口（本轮不修）

产品：

1. `1010003758`
2. `1010003851`
3. `1360003728`
4. `1530003621`

根因：

月交/季交/半年交存在于原始费率表附注公式，例如：

- `月交保险费 = 年交保险费 × 0.09`

但不在当前处理后费率结果表，也未进入当前提取链路。

本轮不做无来源补值。

### 3.2 修改位置

文件：`extract_tier_a_rules.py`

在 `extract_pay_frequency()` 中增加过滤：

若文本命中以下定义性脚注特征，则直接返回 `None`：

```python
skip_markers = [
    "保险费约定支付日",
    "本合同生效日之后每月",
    "每月、每季、每半年或者每年",
    "保险合同生效日在每月或每年",
]
if any(marker in compact for marker in skip_markers):
    return None
```

要求：

1. 过滤只针对上述定义性脚注
2. 不要误伤真正的产品交费频率声明

### 3.3 验收预期

`交费频率`：

1. `1010003676`：从 mismatch 变 hit
2. `1010003677`：从 mismatch 变 hit
3. `1310003720`：不再误提取条款脚注，但由于当前来源仍缺 `趸交`，可保留 mismatch 或进入待后续多来源修复

整体预期：

- 当前：`30%`
- 本轮目标：先提升到 `50%` 左右即可
