# 拆解开发方法论 V1

生效日期：2026-03-28

## 目的

本方法论文档用于统一保险产品智能拆解项目的开发原则、评测口径和问题处理方式。

本项目的目标不是“整理人工结果”，而是建立一条稳定的、可扩展的、以原始文件为核心输入的拆解链路：

原始文件 -> 自动解析 -> 候选值 -> 规则判定 -> 人工复核 -> 标准值

## 核心原则

### 1. 问题默认可解

测试期内遇到的“难问题”，默认应视为可解问题，而不是不可解问题。

原因：
1. 当前测试产品均为人工已处理过、已得出标准结果的产品。
2. 如果人工可以稳定得出结果，系统原则上也应能通过来源补齐、规则补齐、解析优化逐步达到同等能力。

因此，遇到问题时默认判断应是：
1. 来源未接全
2. 规则未表达清
3. 解析链丢失信息
4. 评测口径未对齐

而不是直接判断为“无法自动化”。

### 2. 困难主要来自两类

当前项目中的困难问题，主要来自以下两类：

1. 来源复杂
   - 条款
   - 产品说明书
   - 费率表
   - 现价表
   - 投核保规则
   - 公司材料习惯差异

2. 逻辑复杂
   - 可选责任
   - 条件成立才适用
   - 多文件联合判定
   - 章节编号特殊
   - 负向确认
   - 同一字段多种表达方式

处理问题时，必须先判断它属于来源问题还是逻辑问题，避免混合修复。

### 3. 优先扎实解决更多问题

开发优先级应优先选择“解决一类问题”，而不是“只修一个样本”。

优先顺序建议：
1. 能覆盖一类产品
2. 能覆盖一类文件类型
3. 能覆盖一类字段模式
4. 能稳定进入主链路

不优先做：
1. 只对单个产品写死特判
2. 只在人工页里把结果改对，但不进入正式拆解链

### 4. 修样本必须抽象为系统能力

每次修复一个产品时，都必须同步回答三个问题：

1. 这个问题属于哪一类问题？
   - 例如：附录碎片化 blocks、费率表 PDF、说明书兜底、跨 block 断裂、多来源字段等

2. 这次修复是否可以抽象成通用规则？
   - 若可以，应优先沉淀为通用规则
   - 若暂时不可以，也应明确记录为哪类特例

3. 这次修复能顺带覆盖哪些类似产品？
   - 相同公司
   - 相同文件类型
   - 相同字段模式
   - 相同解析异常

最终目标是形成“能力集合”，而不是“补丁集合”。

### 5. Codex 不接触阿里云数据库

为保证拆解测试真实性，Codex 禁止接触阿里云数据库。

具体约束：
1. 不直连阿里云数据库
2. 不读取在线结果表
3. 不使用目标产品的数据库答案进行开发
4. 只使用本地文件、本地导出 JSON、本地评测结果和用户明确提供的静态材料

如需数据库信息，只能由用户或 Claude 查询后，以静态结果形式提供。

### 6. 原始文件优先

系统的核心处理对象必须是原始文件，而不是人工加工后的文件。

允许的主输入：
1. 原始 PDF
2. 原始 Excel
3. 原始扫描件/图片
4. 原始规则文件

允许的中间产物：
1. md
2. blocks
3. OCR 结果
4. 结构化候选

这些中间产物必须由系统从原始文件自动生成，而不是人工整理后提供。

### 7. 人工加工文件只作验证，不作主输入

人工处理后的文件可以用于：
1. 结果核对
2. 问题定位
3. 特殊阶段过渡验证

但不能作为主输入来源。

如果项目长期依赖人工加工文件：
1. 项目价值会大幅缩水
2. 无法证明真实拆解能力
3. 无法支撑规模化自动化
4. 无法支持未来的反向生成目标

因此，任何依赖人工整理文件才能成立的能力，都不能视为正式拆解能力。

### 8. 答案隔离

对目标产品进行拆解测试时：

可用：
1. 原始资料包
2. 通用标准库
3. 标准值参考库
4. 病种标准库
5. 枚举/映射规则

不可用：
1. 该产品在结果库中的最终标准值
2. 该产品的答案镜像 JSON
3. 该产品的病种数量答案汇总

允许但需说明：
1. 基于原始文件重新生成的中间识别结果
2. 例如：病种识别明细 -> 病种数量

原则：
允许使用中间识别结果，不允许直接使用目标产品最终答案值。

## 开发与评测要求

### 1. 所有新能力都应尽量进入主链路

优先进入：
1. `extract_tier_a_rules.py`
2. `merge_candidates.py`
3. `eval_tier_a.py`
4. `run_blind_test_v1.py`
5. `eval_blind_test_v1.py`

不建议长期停留在：
1. 人工审核页手改
2. 单次临时 JSON
3. 只存在于对话中的规则

### 2. 评测必须区分“真实提升”和“口径变化”

每次评测变化都要分清：
1. 规则能力提升
2. 来源补齐
3. 样本范围变化
4. 答案隔离边界变化
5. 评测口径调整

否则评测结果不可比较。

### 3. 问题处理顺序

建议顺序：
1. 确认来源是否充分
2. 确认是否属于已知解析异常
3. 确认能否抽象成通用规则
4. 再决定是否编写特例

默认不优先写产品级硬编码。

## 适用范围

本方法论文档适用于：
1. Tier A 字段拆解
2. blind test 修复
3. review task 构建
4. 人工复核页设计
5. 后续病种识别、费率表解析、投核保规则接入等扩展任务
