# 条款拆解主链路技术方案 V2

## 1. 目标

本文档用于明确“保险条款拆解”一期正式主链路，作为对内协同和开发实现的统一口径。

本方案仅覆盖：

- 条款 PDF 接入
- 文档转 Markdown
- Markdown 切块
- 字段候选抽取
- 人工复核
- 正式入库

本方案不覆盖：

- 费率表拆解
- 现金价值表拆解
- 跨资料自动补值
- 模型直接写正式库

---

## 2. 总体链路

正式链路定义如下：

`条款 PDF -> 文档转换服务 -> Markdown -> 切块与字段抽取服务 -> 候选结果 -> 人工复核 -> 正式入库`

该链路必须满足以下原则：

- 文档转换层只做格式转换，不做业务结论
- 抽取层只输出候选结果，不直接写正式库
- 正式结果只能由统一复核入库链路产生
- 所有结果必须可解释、可复核、可追溯

---

## 3. 分层职责

## 3.1 条款 PDF 接入层

职责：

- 接收条款 PDF 上传
- 记录文档元数据
- 生成 `document_id`
- 维护文档版本
- 发起后续转换任务

输出：

- `document_id`
- 文档基础信息
- 当前处理状态

说明：

- 本层不做字段抽取
- 本层不做标准值判断
- 本层不做正式入库

建议接口：

- `POST /api/v1/clause/upload`

---

## 3.2 文档转换层

本层采用 [doc_to_md](https://github.com/ferryhe/doc_to_md) 作为独立文档转换服务。

定位：

- 负责将原始条款 PDF 转为 Markdown 中间文件
- 不负责任何责任项、标准值、算法、正式结果判断

建议引擎策略：

- 默认主引擎：`opendataloader`
- 兜底重跑引擎：`mistral`

使用原则：

- 正式主链路默认只跑一个主引擎
- 主引擎失败或质量不达标时，再人工触发兜底引擎重跑
- 不允许多引擎并行输出同时参与正式结果判断

输出内容：

- Markdown 文件
- 转换状态
- 引擎名称
- 错误信息
- 质量标记

中间文件命名建议：

- `{product_id}_clause_pdf.md`

建议接口：

- `POST /apps/conversion/convert`

---

## 3.3 切块与字段抽取层

本层对外可以设计为一个服务，但内部必须拆为两个模块：

1. `segment`
2. `extract`

原因：

- 对外接口可简化
- 对内保留问题定位能力
- 便于区分是 md 质量问题、切块问题还是抽取问题

### 3.3.1 切块模块

输入：

- Markdown 文件

输出：

- `block_id`
- `page`
- `block_type`
- `title_path`
- `text`
- `char_count`

作用：

- 为规则抽取提供稳定输入
- 为模型抽取提供上下文边界
- 为复核页提供证据定位能力

### 3.3.2 字段抽取模块

输入：

- 结构化 blocks

输出：

- `coverage_id`
- `coverage_name`
- `value`
- `confidence`
- `status`
- `block_id`
- `evidence_text`
- `extract_method`

要求：

- 只输出候选结果，不输出正式结论
- 条款内不自足字段不得猜值
- 必须保留证据块和页码
- 必须保留规则/模型来源

建议对外接口：

- `POST /api/v1/clause/extract`

该接口内部完成：

- md 读取
- block 切分
- 字段候选抽取

---

## 3.4 候选结果层

本层属于 AI 中间层，不属于正式业务结果层。

职责：

- 保存本次抽取任务
- 保存每个字段的候选值
- 保存证据、置信度、来源方式
- 支持重跑、对比、回放、复核

必须保存的信息：

- 任务 ID
- 文档 ID
- 产品 ID
- 字段 ID / 字段名
- 候选值
- 状态
- 置信度
- 证据块
- 抽取方式
- 规则/模型版本

说明：

- 正式业务表不能兼作 AI 中间过程表
- 所有中间结果必须先落候选层，再进入复核

建议接口：

- `GET /api/v1/clause/candidates/{task_id}`

---

## 3.5 人工复核层

职责：

- 查看候选结果
- 查看原文证据
- 接受、修改、驳回候选值
- 形成唯一正式业务结论

页面建议：

- 左侧：原文页码与证据块
- 右侧：字段候选值、状态、抽取方式、置信度

操作建议：

- 接受
- 修改
- 驳回
- 标记无法从条款抽取

说明：

- 一期默认不做盲目全自动
- 即使后续支持 `auto_pass`，也必须保留统一复核链路

建议接口：

- `POST /api/v1/clause/review/submit`

---

## 3.6 正式入库层

职责：

- 仅接收复核通过结果
- 通过统一入库服务写入正式业务库

必须遵守：

- 模型不能直接写库
- 页面不能绕过服务直接写库
- 不允许多个入口产生不同正式结论

建议接口：

- `POST /api/v1/clause/sync`

---

## 4. 正式状态设计

建议正式状态至少包括：

- `uploaded`
- `convert_running`
- `convert_failed`
- `converted`
- `degraded`
- `extract_running`
- `candidate_ready`
- `review_required`
- `cannot_extract_from_clause`
- `review_passed`
- `synced`

说明：

- `degraded`：Markdown 质量差，但保留中间结果供排查
- `review_required`：需要人工裁决
- `cannot_extract_from_clause`：条款内信息不足，不能形成正式字段值

这些是正式业务状态，不是临时调试标记。

---

## 5. 对 doc_to_md 的正式定位

`doc_to_md` 在本项目中的正式定位：

- 文档预处理层组件

只负责：

- PDF -> Markdown
- 记录转换状态
- 支持主备引擎切换

不负责：

- 责任项识别
- 标准值判断
- DB 字段映射
- 正式入库

因此不能将 `doc_to_md` 视为“条款智能拆解系统”，只能视为“上游文档转换服务”。

---

## 6. 推荐实施顺序

建议按以下顺序推进：

1. 完成 `PDF -> Markdown`
2. 完成 `Markdown -> blocks`
3. 完成 `blocks -> 候选字段`
4. 完成候选结果存储
5. 完成人工复核页
6. 最后接正式入库

不建议一开始同时做：

- 产品分类
- 责任树激活
- 标准值全链路
- 算法推荐
- 回写旧系统联调

一期先把 clause-only 主链路做稳。

---

## 7. 当前推荐结论

当前正式建议如下：

- 文档转换服务采用 `doc_to_md`
- 默认引擎采用 `opendataloader`
- 异常重跑引擎采用 `mistral`
- `blocks + extract` 对外合并为一个服务，对内保留两层模块
- 所有抽取结果先进入候选层
- 所有正式结果必须经过人工复核

一句话总结：

`doc_to_md` 负责把 PDF 转成可处理 Markdown；切块与抽取服务负责输出可复核候选；人工复核负责形成唯一正式业务结论。`
