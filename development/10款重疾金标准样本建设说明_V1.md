# 10款重疾金标准样本建设说明 V1

## 1. 目标

在条款智能拆解项目启动阶段，优先建设一套可直接用于开发、评测、复核和验收的金标准样本集。

本样本集以桌面 `10款重疾` 原始文件夹和临时数据库中的已拆解结果为基础，形成“原始条款 -> Markdown -> 标准拆解结果”的一一对应关系，为后续规则开发、模型接入、准确率评估提供统一基线。

## 2. 建设原则

1. 样本必须来自真实业务产品，不使用演示数据替代。
2. 一份产品只能保留一套正式金标准，不允许多版本并行成为正式依据。
3. Markdown 仅作为中间解析产物，不作为业务真相。
4. 金标准结果必须来自临时数据库中的既有标准拆解结果，并经过人工核对。
5. 所有样本必须可追溯到原始文件、产品 ID、拆解结果来源和复核人。

## 3. 样本范围

当前一期样本范围固定为桌面 `10款重疾` 文件夹中的 10 款重疾险产品。

样本包含以下三类核心对象：

1. 原始条款 PDF
2. `doc_to_md` 转换后的 Markdown 文件
3. 临时数据库中对应产品的标准拆解结果

## 4. 样本建设目标结构

每个产品样本都必须形成以下 4 层映射：

1. `product_id`
2. 原始文件 `clause_pdf`
3. Markdown 中间文件 `md`
4. 金标准拆解结果 `gold_result`

建议形成统一清单文件，例如：

```json
{
  "product_id": "1134",
  "product_name": "国华康佑保B终身重疾计划",
  "clause_pdf_path": "/Users/zqf-openclaw/Desktop/10款重疾/1134-国华康佑保B终身重疾计划/1134-国华康佑保B终身重疾计划-条款.pdf",
  "md_path": "/Users/zqf-openclaw/codex-openai/development/data/md_cache/1134_clause_pdf.md",
  "gold_result_path": "/Users/zqf-openclaw/codex-openai/development/data/gold/1134_gold.json",
  "status": "ready"
}
```

## 5. 建设步骤

### 5.1 步骤一：建立产品主清单

目标：确定 10 款产品的唯一映射关系。

需要完成：

1. 确认 10 款产品的 `product_id`
2. 确认产品名称与目录名称一致
3. 确认每款产品对应的条款 PDF 文件路径
4. 形成 `sample_manifest` 主清单

输出：

1. `sample_manifest.json` 或 `sample_manifest.csv`

### 5.2 步骤二：生成 Markdown 中间文件

目标：将 10 份条款 PDF 全部转换为 Markdown 中间文件。

规则：

1. 使用 `doc_to_md`
2. 一期解析引擎固定为 `markitdown`
3. 输出目录固定为 `~/code/development/data/md_cache/`
4. 文件命名固定为 `{product_id}_{document_type}.md`
5. 扫描型 PDF 一期不进入自动链路，直接标记 `cannot_extract`

输出：

1. 10 份 Markdown 文件
2. 每份文件的转换状态
3. 失败原因记录

### 5.3 步骤三：导出临时数据库金标准结果

目标：把临时数据库中的既有拆解结果导出为统一结构。

每款产品至少导出以下字段：

1. `product_id`
2. `coverage_id`
3. `coverage_name`
4. `standard_content`
5. 数据来源表
6. 导出时间
7. 导出人

输出：

1. 每产品一份 `gold_result.json`
2. 或一份总表 `gold_result_all.json`

### 5.4 步骤四：人工核对金标准

目标：确认数据库中导出的结果可作为一期正式标尺。

核对范围：

1. `product_id` 是否正确
2. 标准项是否属于当前有效范围
3. 是否包含“招行”前缀无效字段
4. 标准值是否明显缺失或口径异常
5. 是否存在一产品多版本结果冲突

输出：

1. `gold_review_status`
2. 错误项清单
3. 是否允许进入一期评测

### 5.5 步骤五：建立一期标准项白名单

目标：明确一期真正参与拆解和评测的字段。

白名单来源：

1. 当前有效 `cmb_coverage`
2. 重疾险高频标准项
3. 10 款样本中稳定可核对字段

建议白名单先覆盖：

1. 投保年龄
2. 保险期间
3. 交费期间
4. 等待期
5. 重疾赔付次数
6. 中症赔付次数
7. 轻症赔付次数
8. 身故责任
9. 豁免责任
10. 少儿特疾责任
11. 成人特定疾病责任
12. 可选责任
13. 重疾分组情况
14. 首次赔付比例
15. 后续赔付比例

输出：

1. `coverage_whitelist_v1.json`

## 6. 建议目录结构

一期建议在 `development` 下建立如下目录：

```text
development/
|-- data/
|   |-- md_cache/
|   |-- gold/
|   |-- manifests/
|   `-- review/
|-- 10款重疾金标准样本建设说明_V1.md
```

目录用途：

1. `md_cache/`：存放 `doc_to_md` 输出 Markdown
2. `gold/`：存放导出的金标准结果
3. `manifests/`：存放样本清单、白名单、映射关系
4. `review/`：存放人工核对记录

## 7. 建议数据文件

### 7.1 `sample_manifest.json`

记录产品与文件映射关系。

### 7.2 `coverage_whitelist_v1.json`

记录一期参与拆解与评测的标准项白名单。

### 7.3 `gold_result_all.json`

记录 10 款产品统一导出的金标准拆解结果。

### 7.4 `gold_review_report.md`

记录人工核对结论、问题项、处理建议。

## 8. 验收标准

样本集建设完成后，必须满足以下条件：

1. 10 款产品都有唯一 `product_id`
2. 10 款产品都能定位到原始条款 PDF
3. 非扫描型 PDF 已成功转为 Markdown
4. 每款产品都有对应的金标准拆解结果
5. 金标准结果经过人工核对
6. 一期白名单已确认
7. 样本集可直接用于后续开发与评测

## 9. 分工建议

建议最小分工如下：

1. 开发：生成 Markdown、整理目录、建立清单文件
2. 数据/开发：从临时数据库导出金标准结果
3. 业务/拆解同事：核对金标准结果是否可作为一期标尺
4. 项目负责人：确认白名单和最终验收口径

2. 编写任务接口与上传接口设计
