# 人工复核页面设计 V1

版本说明：V1 只做主链路必需功能，不做差异对比、多文件联动预览、反馈闭环分析。这些功能留给 V2。

---

## 1. 页面目标

- 让审核员能快速看到：这个字段的候选值是什么、从哪来的、原文是什么
- 让审核员能操作：接受 / 修改 / 驳回
- 让入库动作只发生在人工确认之后

---

## 2. 页面结构

三栏布局：

```
┌──────────────────────────────────────────────────────────────┐
│  左栏（字段列表）  │  中栏（字段详情 + 操作区）  │  右栏（原文预览）  │
│  ~240px            │  flex-1                      │  ~380px           │
└──────────────────────────────────────────────────────────────┘
```

### 2.1 顶部任务栏（全宽）

| 元素 | 说明 |
|------|------|
| 产品名称 | 如"招商信诺真惠保重大疾病保险（互联网专属）" |
| 产品 ID | product_id |
| 任务 ID | task_id，可点击复制 |
| 整体进度 | "已审 N / 共 M 项" |
| 提交入库按钮 | 所有字段全部 `accepted` 或 `rejected` 后才可点击 |

---

### 2.2 左栏：字段列表

每行一个字段，显示：

- 字段名（coverage_name）
- 当前状态标签：
  - `待审核`（review_required）— 灰色
  - `已接受`（accepted）— 绿色
  - `已修改`（accepted，有 reviewer_value）— 蓝色
  - `已驳回`（rejected）— 红色
  - `无法提取`（cannot_extract / cannot_extract_from_clause）— 橙色
- 点击后右侧显示该字段详情

状态标签显示规则：
- 只要 `status` 还是 `review_required`，就显示"待审核"
- 审核员点"接受"或"修改"后变"已接受"或"已修改"
- 点"驳回"后变"已驳回"
- `cannot_extract*` 状态不需要人工操作，标注后跳过即可

---

### 2.3 中栏：字段详情 + 操作区

#### 顶部：字段元信息

```
字段名：投保年龄
来源类型：clause（条款 PDF）
来源文件：700-招商信诺真惠保重大疾病保险（互联网专属）-条款.pdf
提取方法：rule: age_pattern
置信度：0.95
```

#### 主体：三值展示

| 标签 | 字段 | 说明 |
|------|------|------|
| 原始值 | raw_value | 从文档提取的原始表达，不做规范化 |
| 规范化值 | normalized_value | 经规范化后的候选标准值 |
| 最终入库值 | final_value（可编辑） | 默认等于 normalized_value，审核员可修改 |

说明：
- raw_value 和 normalized_value 只读
- final_value 是可编辑输入框，审核员修改后自动勾选"已修改"

#### 底部：操作按钮 + 备注

```
[ 接受 ]  [ 修改并接受 ]  [ 驳回 ]

审核备注（可选，文本框）：_______________
```

操作说明：
- **接受**：final_value = normalized_value，status → accepted
- **修改并接受**：final_value 由审核员填写，status → accepted（带 reviewer_value 标记）
- **驳回**：final_value 置空，status → rejected，备注必填

---

### 2.4 右栏：原文预览

显示该字段的证据文本块：

```
[ 来源：条款 PDF 第 7 页 ]

23 保险费约定支付日：为本合同生效日之后每月、每季、每半年或者
每年（根据本合同约定的交费方式确定）的对应日。该月如无对应日，
则以该月最后一日为对应日。

block_id: 1010003851_p7_b219
```

显示规则：
- 展示 evidence 中的 text
- 展示 page（页码）
- 展示 block_id（可选，折叠显示）
- 若 evidence 有多个 block，全部展示，按 page 升序排列
- 原文本中高亮命中关键词（前端做简单字符串匹配）

---

## 3. 状态流转

```
review_required
    ├── 接受     → accepted
    ├── 修改接受 → accepted（有 reviewer_value）
    └── 驳回     → rejected

cannot_extract / cannot_extract_from_clause
    → 不需要操作，页面标注跳过

accepted / rejected
    → 可以重新点按钮反悔（允许撤回）
```

最终入库条件：所有 `review_required` 字段已处理完毕（无遗留待审），点击"提交入库"。

## 4. 接口对接（V1 最小集）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/clause-parse/tasks/{task_id}` | GET | 获取任务 + 全部候选结果 |
| `/api/v1/clause-parse/review` | POST | 提交单字段审核结果 |
| `/api/v1/clause-parse/review/batch-import` | POST | 提交入库（所有 accepted 字段） |

#### GET 任务结果返回结构（V1 最小）

```json
{
  "task_id": "task_001",
  "product_id": "1010003851",
  "product_name": "招商信诺真惠保重大疾病保险（互联网专属）",
  "task_status": "review_required",
  "items": [
    {
      "candidate_id": "cand_001",
      "coverage_name": "投保年龄",
      "raw_value": "0-65周岁",
      "normalized_value": "0（28天）-65周岁",
      "final_value": null,
      "extract_method": "rule: age_pattern + domain_default_28d",
      "confidence": 0.95,
      "status": "review_required",
      "source_type": "processed_rate",
      "source_file": "700-1-招商信诺真惠保重大疾病保险（互联网专属）-费率解析结果表.xlsx",
      "evidence": [
        {
          "page": 7,
          "block_id": "1010003851_p7_b219",
          "text": "..."
        }
      ]
    }

```

#### POST 提交审核

```json
{
  "candidate_id": "cand_001",
  "action": "accept",
  "final_value": "0（28天）-65周岁",
  "comment": ""
}
```

action 枚举：`accept` / `reject`

## 5. V1 不做的内容（留给 V2）

| 功能 | 原因 |
|------|------|
| 判定链路逐层展开（raw→md→block→normalized） | 需要 logic_trace_json 完整填充，当前中间结构还在演进 |
| 多来源差异对比（clause vs processed_rate vs 说明书） | 多来源接入尚未稳定 |
| 多文件联动预览（PDF 内嵌跳转） | 工程成本高，V1 展示文本即可 |
| 反馈闭环统计与分析 | 需要积累数据后才有意义 |
| 批量产品列表视图 | V1 先做单产品入口 |

---

## 6. 实现说明

- 前端框架不限，与现有 AIX 项目保持一致
- 右栏原文展示 V1 纯文本即可，不需要 PDF 内嵌渲染
- 所有审核操作实时通过 POST /review 保存，不依赖整页提交
- 提交入库接口在后端做二次校验：所有 `review_required` 必须已处理，否则返回 400
- 入库写入只走统一校验服务，不允许页面直接写 cmb_product_coverage

---

## 7. 验收标准

- [ ] 能打开任意一个 task_id 的复核页
- [ ] 字段列表状态实时更新（操作后即刻刷新左栏标签）
- [ ] 接受 / 修改 / 驳回三个操作均可执行且持久化
- [ ] 原文证据显示正确（block_id、page、text 三者对齐）
- [ ] 提交入库按钮在有未审字段时禁用，全审后激活
- [ ] 后端入库接口有防护：未通过审核的字段不能写入正式表
