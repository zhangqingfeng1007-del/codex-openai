# 费率标准化设计 V1

生成时间：2026-03-25

## 一、目标

本模块用于承接“处理后费率表”并输出可直接贴近 `cmb_product_rate` 的标准化行结构。当前阶段不重建原始 PDF 费率识别能力，优先复用历史口径和现有处理后结果。

## 二、范围

当前只处理：

1. 处理后费率表 `.xlsx`
2. 输出对齐 `cmb_product_rate` 的核心字段
3. 补充来源信息、单位信息和基础默认值

当前不处理：

1. 原始 PDF / Excel 费率表识别
2. 费率中转表解析
3. 复杂年金类字段的完整建模

## 三、文件划分

### 3.1 `rate_field_dicts.py`

职责：

1. 迁移旧 `datas.py` 的核心业务字典
2. 统一命名
3. 增加“中文原字段 -> 内部字段 -> DB 字段”映射

重点内容：

1. 公司名称 / 公司编号
2. 保险期间规范化
3. 交费期间规范化
4. 性别、交费方式枚举
5. `final_df_head_3` 表头定义

### 3.2 `parse_processed_rate.py`

职责：

1. 读取处理后费率表
2. 按统一字典规范化字段
3. 输出贴近 `cmb_product_rate` 的行结构

## 四、字段映射

### 4.1 中文列名 -> 内部字段名

当前核心映射：

1. `产品编号` -> `product_id`
2. `投保年龄` -> `age`
3. `缴费期间` -> `payment_years`
4. `保险期间` -> `insurance_period`
5. `性别` -> `gender`
6. `固定保额/份数` -> `amount`
7. `交费方式` -> `pay_type`
8. `费率` -> `rate`

### 4.2 内部字段名 -> DB 字段名

当前对齐 `cmb_product_rate` 的核心编码字段：

1. `product_id` -> `product_id`
2. `age` -> `age`
3. `payment_years` -> `pay_time + pay_time_type`
4. `insurance_period` -> `insure_period + period_type`
5. `gender` -> `gender`
6. `amount` -> `amount`
7. `pay_type` -> `pay_type`
8. `rate` -> `rate`

## 五、输出结构

当前输出按单行费率记录组织，核心字段为：

1. `product_id`
2. `amount`
3. `rate`
4. `age`
5. `gender`
6. `pay_time`
7. `pay_time_type`
8. `insure_period`
9. `period_type`
10. `pay_type`
11. `company_name`
12. `product_name`

并补充：

1. `rate_unit`
2. `source_type`
3. `source_file`

## 六、单位口径

处理后费率表当前按“每千元保额费率”处理。

当前假设：

1. `固定保额/份数` 默认应为 `1000`
2. `费率` 列为每千元保额对应费率
3. 输出时保留原始 `rate` 数值，不在转换层提前换算成年保费

说明：

1. AIX 当前评分链路会基于 `rate` 和保额自行换算年保费
2. 因此标准化层只负责保证费率口径一致，不负责业务评分换算

## 七、设计原则

1. 能用则用：旧 `datas.py` 的字典直接迁移
2. 用则简洁：旧中转表展开和数据库映射逻辑不继承，重写成小函数
3. 当前只接处理后费率，不引入原始费率识别复杂度
4. 所有输出必须可追溯到来源文件

## 八、验证结论

已使用 `1010003676` 的处理后费率表对生产库 `cmb_product_rate` 做字段级比对，验证结果如下：

1. 行数一致：`618 = 618`
2. 编码字段一致：`pay_time`、`pay_time_type`、`insure_period`、`period_type`
3. 费率值一致：`rate` 在容差 `0.001` 内零差异
4. `pay_type` 一致：零差异
5. 当前解析器输出可直接替代该产品的人工上传结果

当前判断：`parse_processed_rate.py` 已从 PoC 升级为已验证模块，可作为处理后费率表进入标准库链路的基础件。
