# AIX智能体研发治理原则（简化版）

> 目标：保留真正影响项目成败的约束，删除过度展开的描述，保证“简洁、有效、可执行”。

## 1. 总原则

1. 模型不是业务真相来源。
2. 正式结论必须来自统一的数据、规则和服务。
3. 高风险场景优先稳定、可控、可审计，不追求“看起来更聪明”。
4. 研发阶段允许模型辅助，但关键规则必须人工确认。
5. 每次开发都必须形成“实现 + 测试 + 审查 + 可回退”的闭环。

## 2. 业务治理原则

1. 产品规则、评分规则、合规话术、数据库字段映射、审计字段、上线阈值，不能由模型自行拍板。
2. 保险推荐、定价、评分必须依赖真实产品数据、精算规则和明确配置，不能靠大模型自由生成。
3. 不允许多个模型在生产链路中各自产生不同的正式推荐结论。
4. 模型可以参与理解、追问、解释、摘要、复核，但不能绕过规则引擎直接输出正式业务结论。

## 3. 架构治理原则

1. 保持统一底座：数据源、精算引擎、评分规则、审计链路必须统一。
2. 界面层只负责输入、展示和交互，不承载不可追踪的核心业务真相。
3. 核心业务逻辑必须可复用、可测试、可替换，不能散落在多个页面或多个模型提示词里。
4. 同一业务规则只能有一个主实现，禁止长期并存多套口径。

## 4. 开发治理原则

1. 先把边界和数据模型想清楚，再写实现。
2. 复杂业务优先抽成清晰的数据结构和计算模块，不要靠页面事件堆逻辑。
3. 修复问题时必须同步删除废弃代码、旧字段、旧分支和无效配置。
4. 不允许为了赶进度留下“临时但已接入主链路”的长期代码。
5. 新增能力如果改变主数据结构，必须同步改文档、测试和导出结构。

## 5. 测试治理原则

1. 测试必须覆盖主链路，不只测演示页、样例页或简化页。
2. 核心计算逻辑必须有可重复的对照样例。
3. 每次修复后至少回归：
   - 正常路径
   - 边界条件
   - 异常输入
   - 导出结果
4. 测试结论必须能回答两个问题：
   - 结果对不对
   - 结构有没有变差

## 6. 审计与可维护性原则

1. 每个正式结果都应能追溯输入参数、计算路径和输出结果。
2. 文档中的规则描述必须能映射到代码；代码中的关键口径必须能映射回文档。
3. 审查时不仅看“是否能运行”，还要看：
   - 是否重复实现
   - 是否存在沉积代码
   - 是否有不可解释的隐式逻辑
   - 是否容易继续扩展

## 7. 模型协作原则

1. 大模型适合做规划、实现、审查、补测、总结，但职责要分开。
2. 主开发模型负责实现，审查模型负责找错和补测，不能让同一个模型既写又直接判定自己没问题。
3. 模型输出必须接受人工确认，尤其是业务规则和上线判断。

## 8. 执行检查清单

每次开发或改版前后，至少检查以下 8 项：

1. 业务真相来源是否唯一。
2. 数据模型是否支持真实业务，而不是只支持演示场景。
3. 核心逻辑是否集中，而不是散落在 UI 事件中。
4. 是否引入了重复口径或双轨逻辑。
5. 修改后是否清理了无用代码。
6. 文档、实现、导出、测试是否同步更新。
7. 主链路是否被真实测试覆盖。
8. 结果是否可解释、可复核、可追溯。

## 9. 这份简化版的使用方式

1. 先用它审文档和方案。
2. 再用它审代码和测试。
3. 最后用它做上线前检查。

如果代码“能跑”，但违反了这些原则，仍然不能算合格交付。
