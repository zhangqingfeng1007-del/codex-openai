# Claude Code 工作安排调整说明与项目目标

## 一、背景

本项目不是一个从零开始、数据缺失的普通 AI 开发项目。我们已经拥有长期治理并在商业化中实际使用的保险产品数据资产，覆盖大部分险种，并具备以下基础：

- 完整的保险产品责任标准库
- 已标准化的新产品数据库
- 历史老数据库中的大量已处理产品
- Excel 管理表中的补充产品数据
- 结构化的产品病种、费率、责任、关系、现金价值、万能/分红/年金等数据

因此，当前项目的关键问题已经不是“有没有数据”或“能不能做”，而是“如何把已有数据资产、精算能力和开发能力组织起来，转化为稳定、可信、可扩展的系统能力”。

## 二、为什么要调整工作安排

过去如果按普通软件项目方式推进，容易出现以下问题：

- 程序员主导所有问题，但很多问题本质上是责任定义、病种口径、精算规则问题
- 已有标准库和历史产品数据没有被充分复用，导致重复建模
- 数据、精算、开发三条线节奏不一致，造成反复返工
- 功能看似在开发，实际主数据和业务口径没有真正统一

因此，工作安排必须调整为：

`数据资产驱动 + 精算规则驱动 + 开发实现 + 集成验收`

## 三、调整工作的主要目的

本次调整的核心目的有四个：

### 1. 把已有数据资产变成正式系统底座

后续开发必须优先复用：

- 产品责任标准库
- 产品主数据库
- 病种与责任拆解数据
- 费率及相关配置数据

不能绕开这些标准数据，另起一套自由口径。

### 2. 把项目从“开发驱动”改成“数据 + 精算 + 开发”共同驱动

很多关键问题不是代码问题，而是：

- 责任定义问题
- 产品标准化问题
- 病种口径问题
- 精算映射问题

所以开发必须建立在标准数据和精算口径上，而不是单独推进。

### 3. 明确按能力边界分工

后续工作不再采用“谁有空谁补一下”的方式，而是按以下边界组织：

- 产品与数据资产决策
- 精算与责任口径
- 系统实现与联调
- 验收与收口

### 4. 先做阶段性闭环，再做更大扩展

当前阶段的优先目标不是同时做全险种、全精算、全图片理解，而是：

- 先把重疾险主链路做稳
- 先把责任标准和产品数据接入做稳
- 先把精算核对和推荐逻辑跑通
- 在此基础上再扩展百万医疗险和更复杂能力

## 四、当前项目的正确定位

当前项目应理解为：

`基于全险种产品主数据平台，先落地重疾险智能体与定价支撑能力，再逐险种扩展。`

这意味着：

- 全险种数据是长期底座
- 一期交付不等于全险种全部开放
- 必须区分“底层数据已具备”和“前台能力已开放”

## 五、后续执行原则

Claude Code 在后续执行任务时，必须遵守以下原则：

1. 优先复用已有标准数据，不自由发明责任定义和产品结构
2. 遇到责任、病种、费率、精算映射问题时，不臆想，按现有数据结构设计
3. 不把“有字段 / 有页面 / 有提示”误判为“能力已完整实现”
4. 所有输出都要便于精算复核、便于 Codex 审核、便于团队留档
5. 当前阶段优先做：
   - 主链路收口
   - 数据接入
   - 精算核对
   - 测试补齐
   - 文档与边界说明一致性
6. 当前阶段不要优先做：
   - 与当前目标无关的大规模功能扩展
   - 脱离标准库的自由建模
   - 过早做全险种铺开

## 六、当前阶段的工作目标

当前阶段的目标不是“证明能开发”，而是：

- 让已有数据资产正式进入系统
- 让责任标准、产品数据、精算能力、推荐能力使用同一套口径
- 让团队形成稳定的协作机制
- 为下一阶段的险种扩展打下统一底座

一句话总结：

`本次工作安排调整的目的，是把公司已经具备的产品数据优势和精算优势，真正转化成一个可用、可信、可扩展的保险智能体系统。`
